## 1. Problem Statement

**Knowledge Graph Construction (KGC)** traditionally done via:
- **Pipeline methods**: NER → relation extraction → linking (error propagation)
- **Seq2seq methods**: Flatten KG into text (loss of structure)

**Problem**: Natural language flattening loses **structural semantics**:
- Triple order arbitrary
- Schema constraints hard to enforce
- Complex graphs become long sequences

**Key insight**: **Code** inherently has structure - better model for structured output:
- Programming languages have explicit syntax
- Code LMs (Codex, CodeLlama) understand structure
- Can generate syntactically valid, structured output

**Hypothesis**: Treat KG construction as **code completion task** using code LM.

---

## 2. Proposed Solution: CodeKGC

Three main innovations:

### 2.1 Schema-Aware Prompting

**Problem**: Standard prompts don't enforce schema adherence.

**Solution**: Encode KG schema into prompt in **code-like format**:

```
# Schema:
# Node types: Person, Organization, Location, Event
# Relations: works_for, located_in, participated_in

# Task: Extract triples from text
# Output format: triple(subject, relation, object)

Text: "Alice works at Google in Mountain View."
Output:
triple(Alice, works_for, Google)
triple(Google, located_in, Mountain_View)
```

**Why it works**:
- Code LMs trained on structured code patterns
- Schema provides template/constraints
- Format resembles function calls (familiar to code LM)

**Prompt design**:
- Include schema definition
- Show few-shot examples
- Specify output format precisely
- Use comments to guide

### 2.2 Rationale-Enhanced Generation

**Problem**: Model may generate triples without reasoning.

**Solution**: Chain-of-Thought (CoT) style:
```
Step 1: Identify entities: Alice (Person), Google (Organization), Mountain View (Location)
Step 2: Identify relations:
  - "works at" → works_for
  - "in" (location) → located_in
Step 3: Form triples:
  triple(Alice, works_for, Google)
  triple(Google, located_in, Mountain_View)
```

**Implementation**:
- Prompt includes "Let's think step by step" instruction
- Ask model to first reason, then output triples
- Parse reasoning to verify triple consistency

**Benefit**: Intermediate rationale improves accuracy, especially for complex sentences.

### 2.3 Code LM as Generator

**Model choice**: Code-specific LMs (Codex, CodeLlama, StarCoder)
- Pre-trained on code corpora (GitHub)
- Understand syntax, structure, indentation
- Better at generating structured output

**vs. Text LM** (GPT-3/4):
- Text LM trained on natural language
- May "drift" into prose
- Less reliable format adherence

**Fine-tuning** (optional):
- Can further tune on KG construction dataset
- Teach model specific schema, domain

## 3. Methodology

### Input
- Text passage
- Schema definition (node types, relations)
- Few-shot examples (2-5)

### Processing
1. **Construct prompt**: Schema + examples + target text
2. **Generate with rationale**:
   - First: CoT reasoning (entities, relations)
   - Then: Structured triple output
3. **Parse output**: Extract triples from code-like format
4. **Post-process**:
   - Normalize entity names
   - Link to existing KG (entity resolution)
   - Validate against schema

### Output
- Set of triples (subject, relation, object)
- Each conforms to schema

## 4. Technical Details

### Prompt Template

```
# Knowledge Graph Schema
Node Types: {type1}, {type2}, ...
Relations: {rel1}: {domain} → {range}, ...

# Examples
Example 1:
Text: "{example_text_1}"
Reasoning: "{reasoning_1}"
Output:
{triple_1}
{triple_2}
...

# Target
Text: "{target_text}"
Reasoning:
Output:
```

**Few-shot examples**: From same domain if possible

### Output Parsing

**Pattern matching**:
```python
import re
triple_pattern = r"triple\(([^,]+),\s*([^,]+),\s*([^)]+)\)"
triples = re.findall(triple_pattern, output)
```

**Validation**:
- Check relation in schema relations
- Check subject/object types match relation's domain/range
- Reject malformed triples

## 5. Evaluation

### Datasets
- **Standard KG datasets**:
  - ACE04/05 (news)
  - SciERC (scientific papers)
  - Few-NERD (low-resource NER)

**Task**: Joint entity and relation extraction (KG construction)

### Baselines
- Pipeline: SpERT, CasRel
- Seq2seq: Text2Triple, CopyMTL
- Unified: UIE, OneIE

### Code LM Models
- Codex (Code-davinci-002)
- CodeLlama (7B, 13B, 34B)
- StarCoder (15B)

### Results

**ACE04** (F1 score):
- CodeKGC (Codex): **52.8%**
- Best baseline (UIE): 48.3%
- Improvement: +4.5% absolute

**SciERC**:
- CodeKGC: **38.7%**
- Baseline: ~35%

**Few-NERD** (low-resource):
- CodeKGC: **+3-5%** over baselines
- Shows robustness with limited examples

### Ablation Studies

| Variant | ACE04 F1 |
|---------|----------|
| Full CodeKGC | 52.8 |
| - No rationale | 50.1 (-2.7) |
| - No schema prompt | 48.9 (-3.9) |
| - Text LM (GPT-3) | 49.2 (-3.6) |
| - No few-shot | 47.8 (-5.0) |

**Findings**:
- Rationale important (+2.7%)
- Schema crucial (+3.9%)
- Code LM better than text LM (+3.6%)
- Few-shot helpful (+5%)

## 6. Key Insights

### 6.1 Structure-Preserving Generation
Code format forces structured output:
- Triple format: `triple(s, r, o)`
- Each triple on separate line
- Indentation for grouping
- Code LM trained to produce valid syntax → fewer format errors

**vs. Natural language**:
- "Alice works for Google" → ambiguous parsing
- `triple(Alice, works_for, Google)` → unambiguous

### 6.2 Schema Integration
Embedding schema in prompt guides generation:
- Model knows valid node types, relations
- Reduces hallucination of invalid relations
- Enforces domain/range constraints implicitly

### 6.3 Rationale as Intermediate Supervision
Chain-of-thought:
- Makes model "show its work"
- Easier to debug errors
- Intermediate steps can be verified
- Improves final output quality

### 6.4 Code LM vs Text LM
Code LMs have **inductive bias** for structure:
- Trained on indentation-sensitive languages
- Understand nesting, parentheses, brackets
- Better at following templates

**Example**: Code LM less likely to output prose accidentally.

## 7. Application to Code Knowledge Graphs

**Direct relevance** to our papers:
- CodeKGC approach can build **code-specific KGs**
- Schema: code entities (functions, classes, modules)
- Relations: calls, imports, defines, inherits
- Few-shot examples: show code→graph extraction

**Advantages over manual KG construction**:
- Automated, scalable
- Adaptable to new codebases
- Can incorporate code comments/docs

**Why this matters**: Papers like 2505.14394, 2406.07003 use manual AST parsing. CodeKGC suggests **LLM-driven** construction may capture higher-level semantics (intent, purpose) that AST misses.

---

## 8. Limitations

1. **Prompt engineering required**:
   - Schema design affects quality
   - Few-shot example selection matters
   - No automated prompt optimization

2. **Code LM availability**:
   - Codex: API-only, not open
   - CodeLlama: Good but smaller than GPT-4
   - Cost/quality trade-off

3. **Schema rigidity**:
   - Fixed schema in prompt
   - Cannot discover new relation types
   - Requires pre-defined ontology

4. **Error propagation**:
   - No reasoning verification
   - Wrong reasoning → wrong triples
   - No backtracking

5. **Scalability**:
   - Process one document at a time
   - No cross-document coreference resolution
   - Entity linking separate step

## 9. Technical Depth: Schema Design

### For Code KGs

**Node Types** (suggested):
```
MODULE
CLASS
FUNCTION
METHOD
FIELD
VARIABLE
PARAMETER
IMPORT
```

**Relation Types**:
```
CONTAINS: MODULE → CLASS/FUNCTION
DEFINES: FILE/CLASS → FUNCTION/METHOD
CALLS: FUNCTION → FUNCTION (direct call)
USES: STATEMENT → VARIABLE/FUNCTION
INHERITS: CLASS → CLASS
IMPORTS: MODULE → MODULE
HAS_PARAMETER: FUNCTION → PARAMETER
HAS_RETURN_TYPE: FUNCTION → TYPE
```

**Type constraints** (domain/range):
- CALLS: FUNCTION → FUNCTION
- CONTAINS: MODULE → {CLASS, FUNCTION, IMPORT}
- HAS_PARAMETER: FUNCTION → PARAMETER

## 10. Comparison to Traditional Code KG Construction

| Aspect | Traditional (AST) | CodeKGC (LLM) |
|--------|-------------------|---------------|
| **Source** | AST parser | Text + comments |
| **Granularity** | Statement/expression | Higher-level concepts |
| **Semantic** | Structural only | Intent, purpose (from docs/comments) |
| **Scalability** | Fast, deterministic | Slower, LLM cost |
| **Accuracy** | 100% on structure | ~90% (model errors) |
| **Coverage** | All code | May miss implicit relationships |

**Hybrid approach possible**:
- AST for structure (calls, contains, inherits)
- LLM for semantics (intent, documentation linking, design patterns)

## 12. Significance for Code Knowledge Graphs

CodeKGC demonstrates that **code LMs can generate structured output** more reliably than text LMs. This has two implications:

1. **KG construction can be LLM-driven** (not just AST):
   - Capture higher-level semantics
   - Include documentation relationships
   - Identify design patterns

2. **Schema-as-code pattern**:
   - Treat KG schema like API contract
   - Enforce via code format
   - Leverage code LM's structural understanding

**In our papers**:
- 2505.14394 uses LLM to generate descriptions (inspired by this)
- SemanticForge could use CodeKGC for schema-compliant triple extraction
- GraphCoder's CCG could be partially LLM-generated (high-level)

For code repository KG:

```python
prompt = f"""
# KG Schema for Code Repository
Node Types:
- MODULE: Python module/package
- CLASS: Class definition
- FUNCTION: Function (standalone or method)
- METHOD: Class method
Relations:
- CONTAINS: MODULE → {{CLASS, FUNCTION}}
- DEFINES: CLASS → METHOD

- USES: STATEMENT → VARIABLE
- INHERITS: CLASS → CLASS

Example 1:
Text: "class Circle(Shape): def area(self): return π * self.radius"
Reasoning: Entities: Circle (CLASS), Shape (CLASS), area (METHOD), radius (FIELD)
Relations: Circle inherits Shape, Circle defines area, area uses radius
Output:
triple(Circle, INHERITS, Shape)
triple(Circle, DEFINES, area)
triple(area, USES, radius)

# Target
Text: {code_text}
Reasoning:
Output:
"""

response = code_lm(prompt)
triples = parse_triples(response)
```

## 14. Conclusion

CodeKGC pioneers **structured generation for KGC** using code LMs. The core idea: **if you need structure, use a model that understands structure**.

For code knowledge graphs, this means:
- We can construct KGs from code/docs using CodeLlama, not just AST
- Capture semantic relationships AST misses
- Schema-guided generation enforces consistency

**Limitations**: LLM errors, cost, scalability. But combined with AST (hybrid) could get best of both.

**Takeaway**: Code LMs are **underutilized** for code KG construction. Most papers still use traditional parsing. CodeKGC suggests LLM-enhanced construction could add valuable semantic layer.

**Open question**: How well does CodeKGC scale to large repositories? Could batch processing + caching make it practical?
