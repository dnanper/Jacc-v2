# EvoCodeBench: An Evolving Code Generation Benchmark Aligned with Real-World Code Repositories

**Paper ID:** 2404.00599v1  
**Authors:** Jia Li, Ge Li, Xuanming Zhang, Yihong Dong, Zhi Jin (Peking University)  
**Venue:** ICLR 2024  
**Date:** March 2024

---

## 1. Problem Statement

Existing code generation benchmarks (HumanEval, MBPP, APPS) have critical misalignment with real-world software development:

- **Standalone functions only**: Benchmarks feature self-contained functions using only built-in/public libraries
- **Missing dependencies**: Real code has intra-class, intra-file, and cross-file dependencies
- **No repository context**: Models generate in isolation, not considering existing codebase patterns
- **Static content**: Risk of data leakage as models train on these benchmarks

**Key Observation**: Analysis of 1M+ functions from 500 real repos shows most code is **non-standalone** - it depends on repository-specific context.

---

## 2. Proposed Solution: EvoCodeBench

A new benchmark that satisfies 5 critical features:

### 2.1 Real-World Alignment

- **Source**: 25 high-quality open-source Python repositories
- **Samples**: 275 function-level tasks
- **Distribution**: Matches real repos in:
  - Code type ratio (standalone vs non-standalone)
  - Dependency count distribution
  - Code complexity metrics

### 2.2 Comprehensive Annotations

Each sample includes:
- **Function Signature**: Target function signature
- **Requirement**: Natural language description of functionality
- **Repository**: Full codebase (hundreds of files)
- **Reference Code**: Ground truth implementation
- **Reference Dependencies**: Categorized as:
  - Intra-class dependencies
  - Intra-file dependencies
  - Cross-file dependencies
- **Test Cases**: For functional correctness validation

### 2.3 Robust Metrics

- **Pass@k**: Standard functional correctness via test execution
- **Recall@k**: Measures dependency generation accuracy

### 2.4 Evolving Benchmark

- Automatic pipeline updates from latest repositories
- Prevents data contamination/leakage
- Scheduled updates (e.g., every 6 months)

### 2.5 Repository-Level Task

**Task Definition**: Given requirement + complete repository, generate target function body.

This simulates actual developer workflow - writing code that integrates into existing codebase.

---

## 3. Benchmark Construction Pipeline

### Stage 1: Repository Selection
- Popular Python repos (>90% Python code)
- Active maintenance, good test coverage

### Stage 2: Sample Extraction
- Identify functions with clear requirements
- Extract non-standalone functions (with dependencies)
- Balance between standalone and context-dependent

### Stage 3: Annotation
- Manual/automatic requirement generation
- Dependency classification
- Test case creation

---

## 4. Key Findings from Evaluation

### 4.1 Performance Gap

| Model | HumanEval Pass@1 | EvoCodeBench Pass@1 |
|-------|------------------|---------------------|
| GPT-4-turbo | ~80% | 20.73% |
| GPT-3.5 | ~65% | ~15% |
| CodeLlama | ~50% | ~10% |

**Critical Insight**: Performance on traditional benchmarks does NOT transfer to real-world repository tasks.

### 4.2 Failure Analysis

Common shortcomings:
1. **Dependency errors**: Cannot correctly invoke existing repository functions
2. **Style inconsistency**: Generated code differs from project conventions
3. **Architecture violations**: Ignores module boundaries, design patterns
4. **Incomplete implementations**: Missing edge cases covered by dependencies

## 5. Technical Contributions

1. **Benchmark Design Principles**: 5 features for real-world alignment
2. **Dataset**: EvoCodeBench-2403 (275 samples, 25 repos)
3. **Task Formulation**: Repository-level code generation
4. **Analysis**: Reveals limitations of current LLMs in practical settings
5. **Open Release**: Full dataset, prompts, model outputs for reproducibility

---

## 6. Limitations & Future Work

- **Limited language support**: Python only (first version)
- **Scale**: 275 samples relatively small compared to HumanEval's 1000+
- **Manual annotation effort**: Requires domain expertise
- **Repository diversity**: Could expand to more domains (web, mobile, systems)

**Future**: Continuous updates, multi-language expansion, integration with KG-based retrieval systems.

---

## 7. Significance

EvoCodeBench establishes a **realistic evaluation standard** for repository-level code generation. It exposes the gap between isolated function generation and practical software development. The benchmark has become a key evaluation dataset for subsequent KG-based code generation papers (e.g., 2505.14394 uses it extensively).

**Takeaway**: Context matters. LLMs need repository understanding, not just general coding ability.
