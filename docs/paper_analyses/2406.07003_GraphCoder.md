**Paper ID:** 2406.07003v2

---

## 1. Problem Statement

Repository-level code completion requires understanding code **context** beyond the current file. Challenges:

- **LLMs lack repository-specific knowledge**: Pre-trained on general code, don't know project-specific APIs, patterns
- **Context modeling**: How to represent "what code is relevant"?
- **Existing RAG**: Uses sequence-based context (last N lines, similar files by embedding)
  - Ignores code's intrinsic structure
  - Cannot capture statement-level dependencies
  - No notion of "importance distance"

**Key insight**: Code has rich **structural relationships** that sequences cannot capture:
- Control flow (what statements can execute before/after)
- Data dependence (what variables are defined/used where)
- Control dependence (what conditions affect this statement)

---

## 2. Proposed Solution: GraphCoder

Graph-based RAG framework with two main innovations:

### 2.1 Code Context Graph (CCG)

**Definition**: Statement-level multi-graph superimposing three graphs:

1. **Control Flow Graph (CFG)**
   - Vertices: Statements and predicates
   - Edges: Sequential execution, jumps, loops
   - From AST: execution order extracted

2. **Data Dependence Graph (DDG)**
   - Vertices: Same statements as CFG
   - Edges: Variable definition → use relationships
   - "Use before define" errors detectable

3. **Control Dependence Graph (CDG)**
   - Vertices: Same statements
   - Edges: One statement's execution controls another's
   - E.g., `if` condition controls statements in its body

**CCG = CFG ∪ DDG ∪ CDG** (multi-graph, edges labeled by type)

**Node attributes**:
- Statement code
- Statement type (assignment, call, return, etc.)
- Containing function/file
- Line number

**Why statement-level?**:
- Finer granularity than function/class level
- Captures intra-procedural relationships
- Enables precise context slicing

### 2.2 CCG Slicing

**Goal**: Extract subgraph relevant to completion target statement.

**Algorithm** (n-hop slicing):
```
Input: CCG G = (V, E), target statement s_target, max_hops h, max_nodes N
Output: Sliced subgraph G_s

Initialize: Q = [s_target], V_s = {s_target}, E_s = {}
while Q not empty:
    v = Q.pop()
    for each neighbor u of v (following edge direction):
        if distance(v, s_target) < h:
            V_s.add(u)
            E_s.add(edge(v, u))
            Q.push(u)
        else: break
Return G_s = (V_s, E_s) with |V_s| ≤ N
```

**Decay-with-distance weighting**:
- Statements closer to target get higher weight
- Exponential decay: weight = exp(-λ × distance)
- Important: Nearby context more relevant than distant

### 2.3 Coarse-to-Fine Retrieval

**Two-stage process**:

**Stage 1: Coarse retrieval by structure**
- Given completion target statement s
- Compute CCG for all candidate contexts in repository
- Slice each CCG (n-hop from s)
- Compare sliced subgraphs by structural similarity
- Retrieve top-k structurally similar contexts

**Stage 2: Fine retrieval by lexical match**
- Within retrieved coarse set
- Re-rank by token/identifier overlap
- Account for naming conventions, specific variables

**Why coarse-to-fine?**
- Structural similarity captures semantic relevance
- Lexical similarity captures surface details
- Combined more accurate than either alone

## 3. Technical Implementation

### Graph Construction

**Per repository**:
1. Parse each file to AST
2. Build CFG from AST (standard algorithm)
3. Build DDG: variable def-use chains
4. Build CDG: from CFG (post-dominator analysis)
5. Merge into CCG (multi-graph)

**Storage**: In-memory graph structures (not Neo4j like 2505.14394)

**Complexity**: O(|V| + |E|) per file. For large repo, built offline once.

### Query Processing

**During completion**:
1. Identify target statement location (cursor position)
2. Extract partial statement (prefix tokens)
3. Build CCG for current file (if not cached)
4. Slice CCG around target (2-3 hops optimal)
5. Search repository CCGs for similar slices
6. Retrieve top-k matching contexts
7. Inject contexts into LLM prompt
8. Generate completion

**Embedding**: Graph neural network or hand-crafted features for similarity:
- Node type distribution
- Control flow patterns
- Variable usage patterns
- Structural graph edit distance

## 4. Evaluation

### Dataset
- **20 real-world Python repositories**
- **8,000 code completion tasks**
- Various locations: mid-function, function start, after import, etc.

### Baselines
- **RAG with BM25**: Traditional sparse retrieval
- **RAG with embeddings**: Dense retrieval
- **NNLM**: Neural network language model (no retrieval)
- **NM-LM**: Neural network with repository token frequency interpolation
- **FT2Ra**: Fine-tuned with repository context

### Results

| Metric | GraphCoder | Best RAG Baseline | Improvement |
|--------|-----------|-------------------|-------------|
| **Code Exact Match** | +6.06 | - | **+6.06** |
| **Identifier Exact Match** | +6.23 | - | **+6.23** |

Both improvements statistically significant.

### Efficiency
- **Less retrieval time**: Graph-based pruning reduces search space
- **Less database storage**: CCG slices more compact than full file embeddings
- **Faster than per-token generation**: Uses per-output generation (single retrieval)

## 5. Key Insights

### 5.1 Structure Over Sequences
Code's structure (control, data, control dependence) is **essential** for context understanding. Sequence-based methods (last N lines, sliding window) miss:
- Variables defined earlier but used later
- Conditions that guard this code
- Call relationships

CCG captures these explicitly.

### 5.2 Granularity Matters
- Function-level graphs (CoCoMIC) miss intra-function relationships
- Statement-level (CCG) captures precise dependencies
- "Context" is not just "surrounding functions" but "statements that influence this one"

### 5.3 Distance Decay
Not all context is equally important:
- Direct data dependencies = highly relevant
- Control predecessor = relevant
- Distant unrelated code = noise
Exponential decay weighting implements this intuition.

### 5.4 Coarse-Fine Synergy
Pure structural search misses naming specifics. Pure lexical misses semantic similarity. Two-stage:
1. Find structurally similar contexts (semantic)
2. Re-rank by lexical match (specifics)

This is more robust than either alone.

## 6. Comparison to GraphCoder vs. Others

| Aspect | GraphCoder | 2505.14394 (KG Gen) | 2408.03910 (CodeXGraph) |
|--------|-----------|---------------------|-------------------------|
| Graph granularity | Statement | Code element (function/class) | Symbol (module/class/function) |
| Graph type | Multi-graph (CFG+DDG+CDG) | General property graph | General property graph |
| Retrieval | CCG slicing + similarity | Hybrid search + n-hop | LLM-generated queries |
| Storage | In-memory | Neo4j | Graph DB |
| Focus | Completion | Generation | General interaction |
| Evaluation | 8K completion tasks | 275 generation tasks | 3 benchmarks |

GraphCoder is **specialized for completion** - understanding immediate context for next token/statement. Others focus on cross-file/module context for generation tasks.

## 7. Limitations

1. **Intra-procedural only**: CCG is within single function/method
   - Does NOT capture cross-file calls (unlike 2505.14394's file-level graph)
   - Limited to current file's structure

2. **Scalability**: In-memory storage may not scale to very large repos (thousands of files)
   - No indexing for cross-file search
   - Would need graph database

3. **Static analysis limits**: Cannot capture:
   - Runtime polymorphism (virtual dispatch)
   - Dynamic imports/reflection
   - Unresolvable indirect calls

4. **No LLM integration for query**: Unlike CodeXGraph, uses fixed graph algorithms
   - Cannot understand natural language queries
   - Only supports completion task

5. **No constraint checking**: Generated code may still violate types/signatures

## 8. Technical Depth: CCG Construction Details

### CFG Construction (from AST)

```
Standard algorithm:
1. Identify basic blocks:
   - Entry/exit nodes
   - Sequences with single entry/exit
   - Split at branches (if, for, while, return, raise)

2. Build control flow edges:
   - Sequential fall-through within block
   - Jump edges between blocks at branches
   - Loop back-edges

3. Predicate nodes:
   - Condition expressions become vertices
   - Links to true/false successor blocks
```

### DDG Construction

```
For each statement s:
    For each variable v defined in s:
        Add v to Def(s)
    For each variable v used in s:
        For all statements t that define v:
            If there's CFG path t → s without redefinition:
                Add edge t → s in DDG
```

### CDG Construction

```
Using post-dominators:
    s control-depends on t if:
    1. There exists path t → s where all paths from t to exit go through s
    2. t is not post-dominated by any node between t and s
```

**Complexity**: O(|V| × |E|) for naive, faster algorithms exist

## 9. Significance

GraphCoder introduced **statement-level structured context** to code completion. Before this, completion used:
- Last N tokens (sequence-based)
- Whole file embeddings
- Function-level retrieval

CCG showed that **fine-grained structural relationships** matter even within a function. The +6 identifier match improvement means GraphCoder better identifies which APIs/variables to use.

**Influence**: This inspired later work (SemanticForge's program dependence graphs, KGCompass's paths) to go deeper than file/function level.

## 10. Future Work

Authors suggest:
- Extend to cross-file context (currently intra-file only)
- Incorporate type information more explicitly
- Handle dynamic language features (reflection, eval)
- Learn graph representations (GNNs) instead of hand-crafted
- Combine with RAG for cross-file dependencies

**Natural extension**: CCG + cross-file call graph = complete repository context. This is what later papers (2505.14394, 2511.07584) attempt.

---

## 11. Conclusion

GraphCoder demonstrates that **code's inherent structure** (control flow, data dependence) is valuable for context modeling. The CCG representation and slicing algorithm provide a principled way to extract "relevant context" based on data/control flow rather than textual similarity.

**Key contribution**: Showing that **intra-procedural structure** alone yields measurable gains (+6 match) over sequence-based methods. This validates the broader hypothesis: code knowledge graphs improve retrieval → better generation.

The limitation (intra-file only) creates opportunity for integration with cross-file KGs from other papers.
