## 1. Problem Statement

LLMs struggle with **repository-scale** code tasks due to:

1. **Long context limits**: Even with 128K+ tokens, large repos exceed context
2. **Retrieval limitations**: Similarity-based misses structural relationships
3. **Task-specific interfaces**: Existing solutions are custom-built for specific tasks
4. **Generalizability**: Hard to adapt system to new code tasks

Current approaches:
- **Similarity retrieval** (BM25, embeddings): Low recall on complex queries
- **Manual tools/APIs**: Task-specific, require expert knowledge, not generalizable
- **Agentless**: Pre-processed skeleton, limited interaction

**Goal**: Create a **unified, flexible system** for LLM-codebase interaction across diverse tasks.

## 2. Proposed Solution: CODEXGRAPH

System architecture with **three components**:

### 2.1 Code Graph Database

**Schema Design** (Python-focused):
```
Node Types:
- MODULE: Python modules/packages
- CLASS: Class definitions
- METHOD: Class methods
- FUNCTION: Standalone functions
- FIELD: Class attributes, instance variables
- GLOBAL_VARIABLE: Module-level variables

Edge Types:
- CONTAINS: Module → Class/Function/Global
- INHERITS: Class → Parent Class
- HAS_METHOD: Class → Method
- HAS_FIELD: Class/Module → Field
- USES: Statement/Expression → Variable/Function used

Node Attributes:
- name, file_path, signature, code (or index), docstring
```

**Two-Phase Construction**:

**Phase 1: Shallow Indexing** (Single-pass)
- Parse each file with AST
- Extract symbols and immediate relationships
- One scan, fast
- **Limitation**: Misses cross-file CONTAINS, complex inheritance

**Phase 2: Complete Relationships** (DFS traversal)
- Depth-first search across files
- Resolve imports to absolute paths
- Build inheritance chains (including multi-level)
- Resolve re-exports (`from x import y` that re-exports from z)
- Handles Python's dynamic import system

**Storage Optimization**:
- Code snippets stored as **indexed references**, not inline
- Graph DB (Neo4j) stores structure + metadata
- Actual code in separate file storage

### 2.2 LLM Agent with Graph Interface

**Primary LLM Agent**:
- Receives user query (natural language)
- Analyzes query to determine needed information
- Generates **natural language sub-queries** for retrieval

**Translation LLM Agent**:
- Takes natural language sub-queries
- Translates to **graph query language** (Cypher-like)
- Example:
  ```
  NL: "Find where LinearClassifier is defined"
  Cypher: MATCH (c:Class {name: 'LinearClassifier'})<-[:CONTAINS]-(m:Module) RETURN m
  ```

**Execution**:
- Graph queries sent to Code Graph Database
- Results returned to Primary Agent
- Primary Agent synthesizes final answer

**Why two agents?**:
- Separation of concerns: analysis vs. technical query formulation
- Primary agent doesn't need to know Cypher syntax
- Translation agent can be smaller, specialized model

### 2.3 Task-Agnostic Design

**Unified schema works for**:
- **Code chat**: Q&A about codebase
- **Code debugger**: Find bug sources
- **Code commenter**: Generate documentation
- **Code generator**: Create new code
- **Unit tester**: Generate tests

No per-task schema changes needed.

## 4. Key Insights

### 4.1 Graph Database as Universal Interface
**Innovation**: Treat graph DB as **API surface** for codebase.
- Unlike file system access (path-based), graph access is **semantic**
- Can query by relationships, not just location
- Example: "Find all functions that use Class X" is simple graph query
- In file system: would need to search all files

### 4.2 LLM Query Translation
**Problem**: LLMs good at natural language, bad at structured query languages.
**Solution**: Use LLM to translate NL → graph queries.
- Similar to Text-to-SQL
- Graph query language simpler than full programming language
- Translation agent focuses on syntax, primary on intent

**Challenge**: Queries may be malformed, return too much/too little
- System needs error handling, query refinement loops
- Paper doesn't detail failure handling

### 4.3 Schema Design Philosophy
**Task-agnostic** but **language-specific**:
- Schema captures Python's module/class/function structure
- For other languages (Java, C++), different node/edge types needed
- But same overall pattern: symbols + relationships

**Extensibility**: Can add node types without breaking existing queries
- E.g., add TEST node type, link to functions they test
- Existing queries unaffected

## 5. Technical Details

### Graph Query Language

**Cypher-like syntax** (not exact Cypher, but similar):
```cypher
# Find class and its methods
MATCH (c:Class {name: 'TargetClass'})<-[:CONTAINS]-(m:Module)
MATCH (c)-[:HAS_METHOD]->(meth)
RETURN m.name, meth.name, meth.signature

# Find functions using a specific variable
MATCH (f:Function)-[:USES]->(v:Variable {name: 'config'})
RETURN f.name, f.file_path

# Multi-hop: Find methods called by a function
MATCH (f:Function {name: 'process'})-[:CALLS*1..3]->(callee)
RETURN DISTINCT callee.name
```

**Query complexity control**:
- Max hop depth (avoid infinite recursion)
- Result limits (LIMIT clause)
- Timeout handling

### Error Handling

Not explicitly detailed, but necessary:
- **Syntax errors**: Translation agent retry with feedback
- **Empty results**: Query too narrow, suggest broadening
- **Too many results**: Add constraints, limit depth
- **Timeout**: Abort long-running traversal

## 8. Limitations

1. **Query quality depends on translation agent**:
   - Poor translation → wrong results
   - No guarantee queries match intent
   - 51% precision baseline (from SemanticForge comparison)

2. **No constraint checking**:
   - Generated code not validated against repository constraints
   - Can still hallucinate types, signatures

3. **Python-focused**:
   - Schema designed for Python
   - Other languages need different schemas
   - No discussion of multi-language repos

4. **Query execution cost**:
   - Deep traversals expensive
   - Need query optimization, caching
   - Not discussed in paper

5. **Error recovery**:
   - What if query returns nothing?
   - How to refine queries iteratively?
   - Paper mentions "query-refinement" but no details

## 10. Technical Depth: Schema Design Decisions

### Why These Node Types?

**MODULE**: Python's import system makes modules first-class entities
- Everything lives in a module
- Imports create cross-module links

**CLASS / METHOD / FUNCTION**: Python's object model
- Classes contain methods (HAS_METHOD)
- Functions may be standalone or methods

**FIELD**: Python's attribute system
- Class attributes, instance variables
- Important for data flow (DDG)

**GLOBAL_VARIABLE**: Module-level constants/configs
- Often used across functions

### Why These Edge Types?

**CONTAINS**: Hierarchical organization
- Module contains classes/functions
- Class contains methods/fields
- Fundamental containment hierarchy

**INHERITS**: Python's OOP
- Critical for understanding method resolution
- Needed for polymorphic call analysis

**HAS_METHOD / HAS_FIELD**: Class internals
- Alternative to CONTAINS for clarity
- Distinguishes "has method" from "contains function"

**USES**: Data/control dependence
- What entities a statement references
- For "find all code using X" queries

**Missing edges**:
- CALLS (function calls) - possibly inferred from USES or separate analysis?
- IMPORTS - possibly captured by CONTAINS relationships

## 12. Significance

CodeXGraph introduces a **unified, task-agnostic architecture** for LLM-codebase interaction. The insight: **graph databases provide structured, queryable interface that's more natural for code than files**.

This pattern has been adopted/adapted by:
- SemanticForge (more sophisticated query policy)
- KGCompass (path-guided queries for repair)
- DomAgent (KG-guided case selection)

The two-agent design (analysis + translation) separates intent from execution, making the system modular and extensible.

**Takeaway**: Instead of building custom retrievers for each task, build a **general graph interface** and let LLM agents use it. This is more maintainable and adaptable.

**Open question**: How does LLM-generated query quality compare to trained policies (SemanticForge's REINFORCE)? Likely inferior but simpler to implement. Trade-off: ease vs. precision.
