## 1. Problem Statement

**Fuzz testing** effectiveness depends on **fuzz drivers** - code that exercises target APIs with generated inputs. Problems:

### 1.1 Manual Driver Writing
- Time-consuming, requires expert knowledge
- Developers may not know best API combinations
- Limits testing coverage

### 1.2 LLM-Based Driver Generation Challenges
- **Compilation errors**: LLM generates syntactically invalid code
- **API misuse**: Wrong order, missing initialization, incorrect types
- **No understanding of usage patterns**: How APIs are typically combined
- **Seed generation**: Need valid initial inputs for mutation
- **Crash analysis**: Distinguishing driver bugs from library bugs

### 1.3 Current LLM-Fuzzers
- PromptFuzz: API mutation combinations, but no contextual understanding
- OSS-Fuzz-Gen: Zero/few-shot, but blind to codebase specifics
- Lack: Repository-specific knowledge, usage patterns, data flow

**Goal**: Fully automated fuzz driver generation with **codebase-aware** LLM.

---

## 2. Proposed Solution: CKGFuzzer

Multi-agent system with **code knowledge graph** integration:

### 2.1 System Architecture

**Three main components + feedback loop**:

```
┌─────────────────────────────────────────────┐
│   Code Knowledge Graph (pre-built)          │
│   - Functions, data structures             │
│   - Call relationships                     │
│   - API usage patterns                     │
│   - Documentation summaries                │
└─────────────────┬───────────────────────────┘
                  │ queries
                  ▼
┌─────────────────────────────────────────────┐
│   Combination Agent                         │
│   - Queries KG for API combinations        │
│   - Identifies co-used APIs                │
│   - Prioritizes high-value combinations   │
└─────────────────┬───────────────────────────┘
                  │ API combos
                  ▼
┌─────────────────────────────────────────────┐
│   Fuzz Driver-Gen Agent (LLM)              │
│   - Generates driver code                  │
│   - Uses KG context                        │
│   - Creates input initialization           │
└─────────────────┬───────────────────────────┘
                  │ fuzz driver
                  ▼
┌─────────────────────────────────────────────┐
│   Compilation Repair Agent                 │
│   - Attempts compilation                   │
│   - Analyzes errors                        │
│   - Queries KG for fixes                  │
│   - Iterative repair                      │
└─────────────────┬───────────────────────────┘
                  │ compiled driver
                  ▼
┌─────────────────────────────────────────────┐
│   Fuzz Engine                              │
│   - Executes driver with seeds             │
│   - Monitors coverage                     │
│   - Mutation feedback                     │
└─────────────────┬───────────────────────────┘
                  │ coverage/crashes
                  ▼
         ┌────────────────┐
         │  Feedback Loop │
         │  - Coverage-guided mutation │
         │  - Crash analysis            │
         └────────┬───────┘
                  │ updates
                  ▼
         Updates knowledge base, mutates API combos, analyzes crashes
```

### 2.2 Knowledge Graph Construction

**Interprocedural Program Analysis**:

**Nodes**:
- Functions (with signature, return type, params)
- Data structures (structs, classes)
- Files/modules
- Global variables

**Edges**:
- **Calls**: Function A calls Function B
- **Defines**: File/class contains function
- **Uses**: Function uses variable/type
- **Reads/Writes**: Memory access patterns
- **Type relationships**: Inheritance, composition

**Additional Metadata** (LLM-generated):
- Function summary (what it does)
- Usage scenarios (when to use)
- Parameter semantics (valid ranges, constraints)
- Example code snippets

**Storage**: Graph database (Neo4j mentioned or similar)

---

### 2.3 API Combination Agent

**Task**: Identify which APIs to combine in fuzz driver.

**Traditional approach**: Try all combinations → combinatorial explosion.

**CKGFuzzer approach**:

1. **Query KG for co-occurrence**:
   ```cypher
   MATCH (f1:Function)-[:CALLED_WITHIN]->(ctx)←[:CALLED_WITHIN]-(f2)
   WHERE f1 ≠ f2
   RETURN f1, f2, count(ctx) as cooccurrence
   ORDER BY cooccurrence DESC
   ```

2. **Find typical usage patterns**:
   - Init → Configure → Execute → Cleanup patterns
   - Error handling patterns (check return values)
   - Resource management (allocate, use, free)

3. **Prioritize**:
   - High co-occurrence = likely valid combination
   - Cover different data types
   - Include edge cases (NULL inputs, boundary values)

### 2.4 Fuzz Driver Generation Agent

**LLM-based generation with KG context**:

**Prompt includes**:
- Target APIs (signatures from KG)
- Typical usage patterns (from KG)
- Related functions (for setup/teardown)
- Type definitions (structs, enums)
- Error handling patterns

**Generated driver structure**:
```c
#include "target_lib.h"

// Global state
static lib_context_t ctx;

// Input structure
typedef struct {
    uint8_t* data;
    size_t len;
} input_t;

// Fuzz driver entry point
int LLVMFuzzerTestOneInput(const uint8_t* data, size_t len) {
    // 1. Initialize (from KG: required setup)
    lib_init(&ctx);
    
    // 2. Parse input (KG: expected format)
    input_t inp = parse_input(data, len);
    
    // 3. Call target API (from combination agent)
    result_t res = lib_process(&ctx, inp.param1, inp.param2);
    
    // 4. Check result (KG: error codes to handle)
    if (res.status != SUCCESS) {
        lib_cleanup(&ctx);
        return 0;
    }
    
    // 5. Additional API calls (from combination)
    lib_validate(&ctx, res.output);
    
    // 6. Cleanup
    lib_cleanup(&ctx);
    return 0;
}
```

**KG provides**:
- Which init/cleanup functions pair with target
- Parameter types, valid ranges
- Error codes to check
- Thread safety requirements

---

### 2.5 Compilation Repair Agent

**Problem**: LLM-generated code may not compile.

**Repair process**:

1. **Attempt compilation**: `clang -c driver.c -o driver.o`

2. **Parse errors**:
   - Missing includes
   - Type mismatches
   - Undefined symbols
   - Wrong signatures

3. **KG query for fix**:
   ```cypher
   // Missing function: find correct signature
   MATCH (f:Function {name: 'lib_init'})
   RETURN f.signature, f.header_file
   
   // Type unknown: find definition
   MATCH (t:Type {name: 'lib_context_t'})
   RETURN t.definition, t.module
   ```

4. **LLM repair**: Provide error + KG info → regenerate/repair

5. **Iterate** (max N attempts)

**Result**: Valid compiled driver

---

### 2.6 Coverage-Guided Mutation

**Standard fuzzing**: Mutate inputs, track code coverage, prioritize novel coverage.

**CKGFuzzer enhancement**:

**API combination mutation**:
- If current combination doesn't increase coverage:
  - Substitute alternative APIs (from KG)
  - Re-order API calls (from KG temporal patterns)
  - Add missing setup/cleanup (from KG)

**Seed generation from data flow**:
- Analyze driver's data flow (from KG)
- Generate seeds exercising different paths
- E.g., if driver parses input field `type`, generate seeds for each enum value

**Feedback to Combination Agent**:
- Which API combos achieved high coverage?
- Update combination priorities

## 3. Evaluation

### Dataset
- **8 open-source libraries**:
  - libpng, libjpeg, zlib, openssl, etc.
  - Various domains: image processing, compression, crypto

### Baselines
- **AFL**: American Fuzzy Lop (gray-box fuzzing, no driver)
- **libFuzzer**: In-process fuzzing with hand-written harnesses
- **PromptFuzz**: LLM-based driver generation (no KG)
- **OSS-Fuzz-Gen**: Google's automated driver generation

### Results

**Code Coverage Improvement**:
- CKGFuzzer: **+8.73% average** over state-of-the-art
- Range: +3% to +15% depending on library

**Bug Discovery**:
- **11 real bugs** found across tested libraries
- **9 previously unreported** (new discoveries)
- Includes security-relevant issues

**Crash Analysis**:
- Manual review workload reduced by **84.4%**
- KG provides context for root cause analysis
- Distinguishes API misuse vs library bugs

## 4. Key Insights

### 4.1 KG for API Usage Patterns
Not just API signatures, but **how APIs are typically used together**:
- Initialization → operation → cleanup sequences
- Error handling idioms
- Thread safety patterns
- Configuration patterns

This "typical usage" knowledge is **empirical** (from existing code), not just from documentation.

### 4.2 Multi-Agent Specialization
- **Combination Agent**: Knows what APIs to call
- **Generation Agent**: Knows how to write code
- **Repair Agent**: Knows how to fix compilation errors
- **Fuzz Engine**: Knows how to test

Each specialized → better than monolithic LLM.

### 4.3 Iterative Refinement
- Not one-shot generation
- Compile → repair loop
- Coverage feedback → combination mutation
- Crash analysis → driver improvement

**Key**: KG updated with findings (knowledge base grows during fuzzing).

## 5. Technical Depth

### Program Analysis for KG

**Interprocedural analysis**:
- Build call graph across functions
- Track data flow (def-use chains)
- Identify entry points (fuzz target functions)
- Discover helper/setup functions

**Tools**:
- LLVM/Clang for C/C++
- Tree-sitter for multiple languages
- Custom analysis for library-specific patterns

### Coverage-Guided Mutation Details

**Standard**: AFL's genetic algorithm
- Mutate bytes/floats/integers
- Keep seeds that increase edge coverage

**CKGFuzzer adds**:
- **API-level mutation**: Swap one API call for another
- **Parameter mutation**: Based on type info from KG
- **Control flow mutation**: Add/remove error checks (from KG patterns)

**Example**:
```
Original: lib_init(&ctx); lib_process(&ctx, data, len);
Mutated:  lib_init(&ctx); lib_configure(&ctx, timeout=1000); lib_process(&ctx, data, len);
```

## 7. Strengths

1. **Full automation**: From code to fuzzing campaign
2. **Repository-aware**: Understands specific library's patterns
3. **Compilable drivers**: Repair loop ensures valid code
4. **Coverage-efficient**: API mutation more targeted than byte mutation
5. **Bug discovery**: Actually found real bugs (9 new)
6. **Crash triage**: 84.4% reduction in manual review

---

## 8. Limitations

1. **Language support**: Currently C/C++ (libFuzzer targets)
   - Could extend to Python (Atheris), Java, etc.
   - KG schema would differ

2. **KG construction cost**:
   - Interprocedural analysis expensive
   - LLM summaries add cost
   - One-time per library version

3. **Coverage metric limited**:
   - Edge coverage may miss deep semantic bugs
   - Some bugs require specific sequences hard to reach

4. **False positives**:
   - Generated drivers may have bugs themselves
   - Crash analysis not perfect

5. **Scalability**:
   - Large codebases → large KG
   - Query latency could become bottleneck

## 9. Technical Details: KG Query Examples

**For API combination**:
```cypher
// Find APIs commonly called together
MATCH (f1:Function)-[:CALLED_IN]->(ctx:Context)-[:CALLED_IN]->(f2)
WHERE f1.name = 'lib_init' AND f2.name <> 'lib_init'
RETURN f2.name, count(*) as frequency
ORDER BY frequency DESC
LIMIT 10
```

**For compilation repair**:
```cypher
// Get function signature
MATCH (f:Function {name: $func_name})
RETURN f.signature, f.defined_in, f.requires

// Get type definition
MATCH (t:Type {name: $type_name})-[:DEFINED_IN]->(f:File)
RETURN f.path, t.definition
```

**For crash analysis**:
```cypher
// Find similar crash contexts
MATCH (f:Function)-[:CALLS]->(crashed_func)
WHERE crashed_func.has_bug = true
RETURN f, pattern of calls
```

## 14. Conclusion

CKGFuzzer demonstrates KG's value beyond code generation/repair:
- **API usage patterns** critical for effective fuzzing
- **Compilation repair** loop ensures valid drivers
- **Coverage-guided mutation** at API level more efficient

**Results**: +8.73% coverage, 11 bugs (9 new), 84.4% crash analysis reduction.

**Pattern**: Code KG → understand usage → generate quality drivers → effective fuzzing

This expands the application space for code knowledge graphs to **security testing**. Shows versatility of KG approach.

**Takeaway**: Code KGs capture **empirical usage patterns** from existing code. These patterns inform not just "what APIs exist" but "how to use them correctly" - crucial for generating valid fuzz drivers.
