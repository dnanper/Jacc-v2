# KGCompass: Repository-Aware Knowledge Graphs for Software Repair

**Paper ID:** 2503.21710v3  
**Authors:** Boyang Yang, Jiadong Ren, Shunfu Jin (Yanshan University), Yang Liu (Nanyang Technological University), Feng Liu, Bach Le (University of Melbourne), Haoye Tian (Aalto University)  
**Venue:** ACM Conference (publication date: October 2025)

## 1. Problem Statement

**Repository-level software repair** faces unique challenges:

### 1.1 Fault Localization at Scale
- Codebases have **thousands of files**, **millions of lines**
- Bug report rarely explicitly names the buggy function
- **Only 32.7%** of SWE-bench Lite issues mention file/function names
- Must link natural language description to code entities

### 1.2 Search Space Problem
- Candidate fault locations: thousands of functions
- Agentless passes **one file location** → if wrong, repair fails
- More candidates help recall but:
  - Inflate context length
  - Increase cost
  - Raise hallucination risk
  - Degrade patch accuracy

### 1.3 Interpretability Gap
- State-of-the-art achieves >60% repair accuracy
- But provides **no transparent reasoning** for:
  - Why this location was chosen
  - How patch was generated
- Black-box decisions limit trustworthiness

**Example**: SymPy issue where pure LLM proposes wrong "multiply" routine because "multiply" appears in report, but actual bug in "_print_MatAdd" only referenced indirectly via PR #14248.

---

## 2. Proposed Solution: KGCompass

Three-phase framework using **repository-aware knowledge graph**:

### Phase 1: Knowledge Graph Mining

**Graph Construction** - unifies two information sources:

**Code Entities** (from static AST analysis):
- Files, Classes, Functions
- Hierarchical containment: file → class → method
- Static dependencies: calls, imports, attribute references

**Repository Artifacts** (from GitHub data):
- Issues: descriptions, comments, metadata
- Pull Requests: descriptions, modified files, commit messages
- Cross-references: Issue mentions PR, PR mentions issue, PR modifies files

**Temporal Filtering** (critical for realism):
- Only include artifacts **pre-dating** issue creation
- Prevents data contamination from "future" information
- Mirrors what developer had at issue-report time
- **Removes 24%** of initially mined artifacts

**Entity Relevance Scoring**:

For each function entity f, compute score R(f):

```
R(f) = α · cos_sim(emb(issue), emb(f)) + (1-α) · lev_sim(text(issue), text(f))

Where:
- emb(): embedding (semantic similarity)
- lev(): Levenshtein similarity (lexical overlap)
- α: weighting parameter (not specified, likely tuned)
```

**Output**: Top 15 functions by relevance score

**Augmentation**: Add up to 5 additional candidates suggested by LLM

**Result**: 20 candidate fault locations total

### Phase 2: Patch Generation

**Input to LLM**:
- Issue description
- Candidate fault locations (function code)
- **Entity paths** from KG (key innovation!)
  - Example: `issue #14814 → PR #14248 → function "_print_MatAdd"`
  - Shows evolutionary/historical context
  - Reveals related discussions, design decisions

**Prompt includes**:
- Source code of candidate functions
- Metadata (file location, signature)
- Relevant paths and relationships
- Request: "Generate patch that fixes issue"

**Output**: Candidate patches (multiple sampled)

### Phase 3: Patch Ranking

**Two-stage validation**:

1. **Reproduction test generation**:
   - Use LLM to create test that should fail on buggy, pass on fixed
   - Ensures patch addresses actual bug

2. **Execution against tests**:
   - Apply candidate patch
   - Run reproduction test + existing regression tests
   - Pass = patch successful

**Selection**: Patch with highest test pass rate

## 2. Key Innovation: Path-Guided Context

**What are entity paths?**
Not just "function X is related to issue" but **the chain of relationships**:
```
Issue #123 mentions "multiply error"
  ↓ (references)
PR #456 that fixed similar issue
  ↓ (modifies)
Function: array.multiply()
  ↓ (calls)
Function: array.scale()
```

**Why paths matter**:
- Provide **causal/evolutionary context**
- Show **how code evolved** to current state
- Include **discussion context** from PR/issue comments
- Capture **design reasoning**

**In example**: Path from issue to PR to function revealed that "_print_MatAdd" should use "getattr" pattern like "_print_Add" - LLM used this to generate robust solution.

## 3. Evaluation

### Dataset
- **SWE-Bench Lite**: 300 tasks (subset of full SWE-Bench)
- 12 Python repositories
- Standard benchmark for repository repair

### Main Results

**Repair Accuracy**:
- **KGCompass + Claude-4 Sonnet**: **58.3%**
- Baseline pure LLM: ~38.7% (implied from 50.8% relative gain)
- **Relative improvement**: **+50.8%** over pure-LLM baseline

**Model-agnostic improvements**:
- Claude-3.5 Sonnet: +30.2%
- DeepSeek-V3: +115.7%
- Qwen2.5 Max: +156.4%

**Fault Location Accuracy**:
- File-level: **83.6%**
- Function-level: **56.0%**
- Top-20 candidates cover: 91.0% file-level, 66.7% function-level ground truth

**Cost**: Only **$0.2 per repair** (Claude-4 Sonnet API costs)

### Ablation Studies

- **Without KG**: Large drop in both location and repair accuracy
- **Without path context**: Repair accuracy drops ~8-10%
- **Without test generation**: Cannot validate patches
- **Top-5 vs Top-20**: Recall drops significantly (20 needed for 91% coverage)

---

## 4. Multi-Hop Analysis

**Critical finding**: **89.7%** of successfully repaired bugs require **multi-hop** (indirect) connections.

- Only ~10% of bugs have direct issue→function link
- Most require traversing: issue → PR → modified file → function
- Or: issue mentions concept → related PR → actual fix location

**Pure LLM limitation**: Cannot perform reliable multi-hop reasoning across thousands of entities.
- Semantic similarity finds direct matches
- Misses indirect pathways
- "Warm start" problem: starting from wrong location

**KG advantage**: Explicit graph structure enables systematic traversal, not just embedding similarity.

## 5. Technical Deep Dive

### Graph Construction Details

**Static Analysis**:
- AST parsing for hierarchical structure
- Call graph extraction (function calls)
- Import analysis (cross-file references)

**Artifact Linking** (template-based regex):
```
Patterns extracted:
- File paths: "xxx.py", "./xxx/", "file:xxx"
- Symbol names: class names, function names in backticks
- Line numbers: "#L123", "line 45"
- Issue/PR numbers: "#123", "PR-456"
```

**Link resolution**:
- Issue #123 → PR #456 if PR references issue
- PR #456 → files changed (git diff)
- Files → classes/functions (AST lookup)
- Build graph paths

**Temporal filtering logic**:
```python
if artifact.created_at <= issue.created_at:
    include_in_graph()
else:
    exclude()  # Prevent peeking at future
```

### Entity Relevance Formula

```
R(f) = α · cos_sim(emb(issue_desc), emb(f.description))
       + (1-α) · lev_sim(normalize(issue_text), normalize(f.name + f.file_context))
```

- `cos_sim`: Captures semantic relevance (e.g., "authentication" ≈ "login")
- `lev_sim`: Captures lexical overlap (exact token matches)
- `α` likely tuned on validation set (not specified)

**Why both?**:
- Semantic: catches paraphrased mentions
- Lexical: catches specific API names, file names
- Combined more robust

### Patch Generation Prompt Structure

```
You are an expert Python developer fixing a bug.

Issue Description:
{issue_text}

Relevant Code Context:
{for each candidate location:
    - File: path
    - Function: signature
    - Code: implementation
    - KG Path: issue → PR #XXX → this function (reasoning chain)
}

Related Discussion (from PR comments):
{pr_comments if available}

Generate a minimal patch that:
1. Fixes the described bug
2. Maintains existing functionality
3. Follows the repository's coding style

Output in unified diff format.
```

## 6. Comparison to Other Repair Systems

| System | Approach | Fault Location | Patch Gen | Accuracy (SWE-Bench Lite) |
|--------|----------|----------------|-----------|---------------------------|
| **Agentless** | Procedural, single-file | 1 candidate | LLM | ~40% |
| **SWE-agent** | Agent, iterative exploration | Multiple | LLM + tools | ~45% |
| **CodePlan** | Planning-based | Multiple | LLM | ~42% |
| **KGCompass** | KG-guided | Top-20 from KG | LLM | **58.3%** |

**Why KGCompass wins**:
- Precise multi-hop localization (83.6% file-level)
- Rich context via entity paths
- Cost-effective (only 20 candidates, not exhaustive search)
- Interpretable reasoning chains

## 7. Strengths

1. **Precise fault localization**: Narrows 1000s → 20 candidates with 83.6% file recall
2. **Interpretable**: Entity paths show reasoning (not black-box)
3. **Cost-efficient**: $0.2 per repair vs. agent iterations ($1-5 typically)
4. **Model-agnostic**: Works with Claude, DeepSeek, Qwen (all improve)
5. **Temporal awareness**: Filters future artifacts, realistic evaluation

---

## 8. Limitations

1. **Python-only**: AST analysis tied to Python syntax
2. **GitHub-specific**: Relies on issue/PR linking (other platforms differ)
3. **Regex extraction**: Template-based artifact linking may miss:
   - Implicit references ("in the previous PR" without number)
   - Code snippets without file paths
   - Cross-repo references
4. **Static analysis only**: No dynamic execution traces
5. **Single-repo focus**: Not designed for cross-repo repair

---

## 9. Technical Insights

### Why Top-20 Works Better Than Top-5 or Top-50

- **Top-5**: Insufficient coverage - only ~60% file-level
- **Top-20**: 91% file-level, 66.7% function-level - sweet spot
- **Top-50+**: Diminishing returns, context inflation, more noise

**Finding**: Need to cover likely locations without overwhelming LLM.

**Tuning**: 15 KG-based + 5 LLM-suggested = balanced diversity

### Multi-Hop Criticality (89.7%)

This statistic is **crucial**:
- Shows most bugs NOT directly linked to issue text
- Pure LLM/similarity methods fail here
- KG's multi-hop traversal essential
- Without graph, would need agent exploration (expensive)

## 11. Significance

KGCompass demonstrates that **repository metadata (issues, PRs) + code structure** in unified KG dramatically improves repair.

**Key insight**: Bugs are not just "in code" but "in history" - previous PRs, discussions, related issues. KG captures this evolutionary context that pure code analysis misses.

**Result**: 58.3% on SWE-Bench Lite is state-of-the-art for **single-LLM open-source** approach (as of Oct 2025).

**Pattern established**:
1. KG construction (code + artifacts)
2. Multi-hop localization (paths, not just nodes)
3. Context augmentation (paths + code)
4. LLM generation + execution validation

This pattern influences subsequent repair work.

## 14. Conclusion

KGCompass solves repository repair's core challenges:
- **Where to fix?** → KG localization (83.6% file accuracy)
- **How to fix?** → Path-guided context (evolutionary reasoning)
- **Did it work?** → Execution validation

The 50%+ improvement over pure LLM demonstrates **structured repository knowledge** is essential. The 89.7% multi-hop statistic proves implicit reasoning insufficient - need explicit graph traversal.

**Takeaway**: For repair, link **issues** to **code** via **history** (PRs). The complete KG includes not just code dependencies but **human knowledge** embedded in issue discussions and PR descriptions.

This is the first repair system to systematically leverage repository artifacts (not just code) via KG. Sets new baseline for interpretable, accurate repair.
