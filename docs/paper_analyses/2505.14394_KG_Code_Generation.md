# Knowledge Graph-Based Repository-Level Code Generation

**Paper ID:** 2505.14394v1  
**Authors:** Mihir Athale (Northeastern University), Vishal Vaddina (Quantiphi Inc.)  
**Venue:** Not specified (arXiv)  
**Date:** May 20, 2025

---

## 1. Problem Statement

LLMs excel at standalone code generation but struggle with **repository-level context**:

- **Architectural ignorance**: Unaware of project-specific patterns, module organization
- **Dependency blindness**: Cannot find or correctly use existing code
- **Style inconsistency**: Generated code differs from project conventions
- **Redundancy**: Duplicate functionality already present

Current RAG approaches (web search, documentation retrieval) are insufficient:
- Lack structural understanding
- Cannot track inter-file dependencies
- No guarantee of consistency with existing codebase

**Goal**: Improve code retrieval quality → better context → better generation

---

## 2. Proposed Solution: 3-Stage KG Framework

### Stage 1: Knowledge Graph Construction

**Parsing**:
- Parse each file using AST
- Extract: Classes (Ci), Methods (Mi), Functions (Fi), Attributes (Ai)
- Set E = {Ci, Mi, Fi, Ai, ...} per file

**Schema Definition**:
```
Node Types: File, Class, Method, Function, Attribute, Generated Description
Relations: defines_class, defines_function, has_method, used_in, has_attribute, has_description
```

**Metadata Enhancement**:
- Extract docstrings/comments
- **LLM-generated descriptions**: For each code element, use LLM to generate semantic description
- Store in graph as "Generated Description" nodes

**Storage**: Neo4j graph database with indexes:
- Full-text indexes: function names, class names, method names, modules
- Vector indexes: documentation embeddings, LLM-generated description embeddings
- Encoder: all-MiniLM-L6-v2

### Stage 2: Hybrid Code Retrieval

**Query Processing**:
1. **Entity extraction**: LLM identifies schema-aligned entities in user query
2. **Query embedding**: Encoder generates query vector

**Initial Retrieval**:
- Full-text search on LLM-identified entities
- Vector similarity search on documentation/descriptions
- Reverse-map to actual code elements (functions, classes)
- Limit to top-k by score

**Subgraph Expansion**:
- n-hop traversal from initial nodes
- Captures dependencies, imports, usages, related functions
- Parameter: number of hops controls context breadth

**Filtering**:
- Generate embeddings for subgraph nodes
- Filter by similarity to query embedding
- Keep top-k most relevant
- Prunes noise, improves efficiency

### Stage 3: LLM Code Generation

**Input to LLM**:
- User query
- Refined subgraph (with relationships)
- Custom prompt emphasizing:
  - Use relationships/dependencies from graph
  - Align with repository architecture
  - Follow coding conventions visible in graph

**Output**: Contextually-aware, dependency-correct code

## 3. Key Technical Choices

### Hybrid Retrieval Design
- **Full-text**: Exact entity matches (function names, class names)
- **Vector**: Semantic similarity for conceptual matches
- **Graph**: Structural relationships for dependency context

### LLM-Generated Descriptions
- Captures "intent" beyond signatures
- Enables semantic search even when names differ
- Example: function `process_data` might have description "normalizes input features"

### Neo4j Storage
- Native graph queries (Cypher)
- Efficient traversals
- Combined full-text + vector search

## 4. Evaluation

### Dataset
- **EvoCodeBench-2403**: 275 samples, 25 repos
- Task: Generate function body given repository context
- Metric: Pass@1 (test-based correctness)

### Results

| Model + Approach | Pass@1 |
|------------------|--------|
| **Proposed KG approach** | |
| GPT-4 | 32.00% |
| GPT-4o | 33.45% |
| Claude 3.5 Sonnet | **36.36%** |
| **Baselines (from EvoCodeBench paper)** | |
| Local File Infilling (GPT-4) | 20.73% |
| Local File Completion (GPT-4) | 17.45% |
| Without Context (GPT-4) | 7.27% |
| CodeXGraph (GPT-4o) | 36.02% (212/275 samples) |

**Improvement**: ~75% relative over no-context baseline

### Ablation Insights
- Hybrid search outperforms pure full-text or pure vector
- n-hop expansion (2-hop optimal) crucial for context
- LLM-generated descriptions improve retrieval recall

## 5. Strengths

1. **Practical architecture**: Uses established tools (Neo4j, MiniLM)
2. **Hybrid approach**: Combines multiple retrieval signals
3. **Scalable**: Graph database handles large repos efficiently
4. **Adaptable**: Schema can be extended (decorators, types mentioned as future)
5. **Multi-LLM compatible**: Works with GPT-4, GPT-4o, Claude

---

## 6. Limitations

1. **Computational intensity**: n-hop traversal + filtering expensive
2. **Latency**: Graph queries add overhead (not quantified in paper)
3. **Schema limitations**: Current schema doesn't capture:
   - Decorators
   - Variable types (only attributes)
   - Configuration files
   - Tests themselves
4. **LLM description cost**: Additional LLM calls during graph construction
5. **No constraint checking**: Generated code may still violate types/signatures

## 9. Significance

This paper presents a **pragmatic, implementable** KG approach that demonstrably improves over baseline retrieval. The 3-stage pipeline (construct → retrieve → generate) has become a standard pattern in subsequent papers.

The key insight: **LLM-generated semantic descriptions enable semantic search even without sophisticated graph algorithms**. By enhancing traditional retrieval with structure, they achieve most benefits of KG approaches without extreme complexity.

**Takeaway**: You don't need cutting-edge algorithms to get value from KGs. Thoughtful combination of existing techniques (AST parsing, embeddings, graph DBs) yields substantial gains.
