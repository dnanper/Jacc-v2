## 1. Problem Statement

Two fundamental failure modes in LLM code generation:

### 1.1 Logical Hallucination
- **Definition**: Incorrect control/data flow reasoning
- **Examples**:
  - Iterating over wrong collection
  - Missing necessary state mutations
  - Violating expected runtime behavior
  - Semantic execution logic errors (not syntax)

### 1.2 Schematic Hallucination
- **Definition**: Structural/interface violations
- **Examples**:
  - Type mismatches
  - Incorrect argument ordering
  - Missing parameters
  - Calls to non-existent functions
  - Visibility errors

**Root Cause**: Current approaches lack **explicit, queryable repository-wide semantics**:
- RAG: Surface similarity only, misses transitive relationships, type constraints
- Encoder models (GraphCodeBERT): Intra-function only, no persistent KG
- Repair-oriented: Post-hoc validation, doesn't prevent hallucinations

## 2. Proposed Solution: SemanticForge Framework

Four algorithmic innovations:

### 2.1 Dual Static-Dynamic Knowledge Graphs

**Novelty**: First to unify compile-time AST with runtime execution traces.

**Static Graph** (from AST analysis):
- Control Flow Graph (CFG)
- Data Dependence Graph (DDG)
- Control Dependence Graph (CDG)
- Type hierarchies, signatures

**Dynamic Graph** (from test execution):
- Actual execution paths
- Runtime types (concrete types at call sites)
- Variable bindings
- Call frequencies

**Reconciliation Algorithm**:
- Automatically merges static and dynamic information
- **Theorem 4**: Provably converges to ground-truth program dependence graph as test coverage increases
- Resolves discrepancies (e.g., polymorphic calls resolved at runtime)

**Impact**: Dynamic augmentation alone → +7.3% Pass@1, -12.4% logical hallucination

### 2.2 Neural Query Language Generation

**Problem**: LLM-generated graph queries (CodeXGraph approach) have only 51% precision.

**Solution**: Train a policy (REINFORCE algorithm) to generate structured graph queries from natural language.

- **Graph-aware reward**: Based on relevance of retrieved subgraph
- **Learned vs. ad-hoc**: Policy learns what graph patterns match query intents
- **Result**: 73% precision vs 51% for LLM ad-hoc generation

**Significance**: Query quality is critical. Poor queries retrieve wrong context, harming generation.

### 2.3 SMT-Integrated Beam Search

**Breakthrough**: Integrate constraint verification **into** generation, not after.

**How**:
- During beam search, for each candidate token sequence:
  1. Extract partially generated AST
  2. Encode type/interface constraints from KG
  3. Call SMT solver (Z3) to check if continuation can satisfy constraints
  4. Penalize/prune candidates violating constraints

**Constraints Verified**:
- Type consistency (variable types match assignments)
- Signature compliance (correct arguments, return types)
- Visibility (access modifiers respected)
- Architectural invariants (from KG)

**Results**:
- Eliminates **52.2% of schematic errors**
- Only **8.3% latency overhead** (incremental solving)
- **2.4s average generation time** vs. 5.7s for CodePlan, 10-50 iterations for agents

### 2.4 Incremental Maintenance Algorithm

**Problem**: Codebases evolve. Rebuilding KG from scratch is O(n).

**Solution**: Incremental update algorithm:
- When code changes, only recompute affected subgraph
- **O(|R|·log n) complexity** (where |R| = change size)
- **Theorem 3**: Maintains semantic equivalence to full reconstruction

**Critical for**: Continuous integration, active development scenarios

## 3. Evaluation

### Dataset
- **REPOKG-50**: Custom benchmark
- 4,250 repository-level tasks across 50 Python projects
- Comprehensive coverage of various repository patterns

### Main Results

| Metric | CodeLlama-34B | + SemanticForge | Improvement |
|--------|---------------|-----------------|-------------|
| Pass@1 | 34.2% | 49.8% | **+15.6% abs.** |
| Schematic Hallucination Rate (SHR) | 30.8% | 14.7% | **-52.2%** |
| Logical Hallucination Rate (LHR) | 36.2% | 23.1% | **-31.8%** |
| Latency | - | 2.4s | - |

### Comparison to Other Paradigms

| System | Pass@1 | Latency | SHR | LHR |
|--------|--------|---------|-----|-----|
| RAG-Code | 40.1% | 2.9s | 30.8% | 36.2% |
| CodePlan | 42.3% | 5.7s | 31.5% | 33.8% |
| **SemanticForge** | **49.8%** | **2.4s** | **14.7%** | **23.1%** |

**Advantages claimed**:
- vs. Agent-based: Deterministic single-pass (2.4s) vs 10-50 iterations; formal guarantees
- vs. Planning: 2.4× faster, 52.2% fewer schematic errors
- vs. RAG: +17.7% Pass@1, -52.3% schematic hallucinations

### Computational Cost

- 47.1% overhead over base CodeLlama
- **BUT**: Eliminates iterative refinement → net savings
- Energy: 187J per task, 23% net savings due to higher Pass@1
- Scales to 500K LOC with sub-linear query complexity (O(n^0.73))

## 4. Key Insights

### 4.1 Constraint Integration Paradigm
The most significant innovation: **constraints during generation, not after**.
- Traditional: Generate → Validate → Repair (may need multiple rounds)
- SemanticForge: Generate WITH constraints → fewer errors, single pass

This is a fundamental shift from "generate then check" to "generate correctly".

### 4.2 Dual Analysis Value
Static analysis gives structure; dynamic gives reality.
- Static: What COULD happen (all possible paths, types)
- Dynamic: What DOES happen (actual executions)
- Combining both catches errors each alone misses

### 4.3 Formal Guarantees Matter
- Theorem 3: Incremental updates preserve semantics
- Theorem 4: Reconciliation converges to ground truth
- SMT solver provides mathematical guarantees

Contrast with LLM confidence scores - here we have proofs.

## 5. Limitations (Acknowledged by Authors)

1. **Comparison challenges**: Agent-based systems hard to compare directly (different metrics, setups)
2. **Benchmark scope**: REPOKG-50 limited to Python, may not generalize
3. **SMT solver limits**: Complex constraints may be undecidable/expensive
4. **Training requirement**: Query policy needs training data (graph queries)
5. **Tooling complexity**: Integrating SMT into beam search is non-trivial

## 6. Technical Depth

### SMT Integration Details

**Constraint Encoding**:
```python
# For partially generated code:
constraints = []
for node in partial_ast:
    if node.type == "Assignment":
        constraints.append(
            type(node.lhs) == type(node.rhs)
        )
    if node.type == "FunctionCall":
        sig = kg.lookup_signature(node.function_name)
        constraints.append(
            len(node.args) == len(sig.params)
        )
        for i, arg in enumerate(node.args):
            constraints.append(
                type(arg) ⊆ sig.params[i].type  # subtype allowed
            )
```

**Incremental Solving**:
- Previous constraints persist across beam steps
- Only add new constraints from new tokens
- Solver state reused, not restarted

## 9. Conclusion

SemanticForge represents the **constraint-aware generation** paradigm. It demonstrates that preventing errors during generation is more effective than fixing them after. The combination of dual-analysis KGs + SMT-guided decoding + learned query planning creates a system that is:
- More accurate (+15.6% abs.)
- More reliable (-50%+ error rates)
- Faster (single-pass vs iterations)
- Guaranteed (formal theorems)

This is the current state-of-the-art in terms of technical sophistication. The challenge is engineering complexity - integrating all these components is non-trivial.

**Takeaway**: The future is constraint-driven synthesis. Pure pattern completion (what LLMs do natively) is insufficient. We need to ground generation in explicit, verifiable constraints from the codebase.
