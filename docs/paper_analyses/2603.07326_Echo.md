## 1. Problem Statement

**Issue reproduction test generation**: Given a bug report (GitHub issue), automatically generate a test case that reproduces the failure.

**Why this matters**:
- Developers spend huge time understanding and reproducing bugs
- Reproduction tests validate fix correctness (fail before, pass after)
- Essential for CI/CD quality gates
- Bug triage requires reproduction

**Challenges**:

### 1.1 Ambiguous Issue Descriptions
- Issue reports often incomplete
- Missing imports, setup code, expected behavior
- May include partial code snippets without context
- Example from paper (Django-12284):
  ```
  "Cannot override get_FOO_display() in Django 2.2+"
  + minimal code snippet, no imports, no test framework
  ```

### 1.2 Repository-Specific Testing
- Each project uses different test framework (pytest, unittest, custom)
- Different conventions for test naming, location, structure
- Specific execution commands (pytest django/test/...)
- Environment setup (database, settings, env vars)

### 1.3 Lack of Reliable Oracle
- How to verify generated test is correct?
- Issue description may not precisely state expected failure
- LLM-based semantic judgment unreliable
- Need fail-to-pass criterion: test fails on buggy, passes on fixed

### 1.4 Execution Setup
- Running tests requires correct commands, environment
- Issues don't provide this information
- Small mismatch → test doesn't run → cannot validate

## 2. Proposed Solution: Echo

Agent system with three key innovations:

### 2.1 Graph-Enhanced Context Retrieval

**Problem**: Traditional retrieval (Agentless's hierarchical LLM file localization) lacks structural understanding.

**Solution**: Use **code graph** for precise context.

**Graph features**:
- Code entities (files, classes, functions)
- Call relationships
- Import dependencies
- Test locations and patterns

**Automatic Query Refinement**:
- Initial query from issue description
- Check if retrieved context sufficient (LLM judgment)
- If insufficient → refine query, retrieve more
- Iterative until context adequacy threshold

**Compact context set**:
- Not all retrieved info sent to generator
- Filter to most relevant
- Reduces noise, improves generation quality

### 2.2 Automatic Test Execution

**Novel feature**: First system to seamlessly integrate test execution into reproduction generation.

**Workflow**:
1. Generate candidate test
2. **Automatically determine execution command**:
   - Analyze repository structure
   - Detect test framework (pytest vs unittest vs custom)
   - Find test runner configuration
   - Construct appropriate command
3. **Execute test** in isolated environment
4. **Capture results**: Pass/fail, error messages, stack traces

**Why important**:
- Validates test actually runs
- Provides execution feedback for refinement
- Enables fail-to-pass validation

### 2.3 Dual-Version Validation with Patch Generation

**Key insight** (from e-Otter++): Use patched version as oracle.

**Process**:
1. Generate candidate test T
2. Generate potential patch P (that would fix the bug)
3. Apply P to codebase → patched version
4. Run T on:
   - **Original codebase**: Should FAIL
   - **Patched codebase**: Should PASS
5. If T exhibits fail-to-pass behavior → valid reproduction
6. If not, use failure analysis to refine T

**Advantage over pure LLM judgment**:
- Objective execution-based criterion
- No need for LLM to semantically match failures
- More reliable validation

### 2.4 Single-Test Focus

**Unlike prior approaches**:
- LIBRO, AEGIS, e-Otter++: Generate multiple candidates, select best
- Echo: Generate **one high-quality** test through iterative refinement

**Rationale**:
- More cost-effective (1 test vs 10-50 candidates)
- Better cost-performance trade-off
- Sufficient for reproduction purpose

## 3. Technical Implementation

### 3.1 Code Graph Construction

**Sources**:
- Static AST analysis (functions, classes, calls)
- Test file identification
- Framework detection (pytest markers, unittest classes)
- Import relationships

**Graph queries for retrieval**:
```cypher
// Find relevant functions
MATCH (f:Function)-[:CONTAINS_IN]->(file)
WHERE f.name CONTAINS $keyword OR f.docstring CONTAINS $keyword
RETURN f

// Find test files and patterns
MATCH (t:TestFile)-[:USES_FRAMEWORK]->(fw)
RETURN t.path, fw.name, t.structure_pattern
```

### 3.2 Automatic Command Detection

**Heuristics**:
- `pytest.ini`, `setup.cfg`, `pyproject.toml` → pytest
- `unittest` imports in tests → unittest
- Custom runner scripts → parse for invocation pattern

**Command construction**:
```python
if framework == "pytest":
    if "django" in repo:
        cmd = "pytest --ds=test_settings"
    else:
        cmd = "pytest"
elif framework == "unittest":
    cmd = "python -m unittest discover"
```

### 3.3 Patch Generation for Validation

**Need**: Generate plausible patch that would fix the issue.

**Approach**:
- LLM generates candidate fix given issue + focal code
- Not required to be perfect (just needs to change behavior)
- Simple fixes often sufficient: return early, raise error, modify condition

**Why this works**:
- Even imperfect patch likely changes test outcome
- Test that fails on both buggy and patched → invalid
- Test that fails only on buggy → valid reproduction

### 3.4 Error Analysis Loop

If test fails fail-to-pass check:
1. **Analyze failure**:
   - Test error before patch? → test itself broken
   - Test passes both? → not reproducing bug
   - Test fails both? → patch too broad
2. **Generate feedback**: "Test needs to check X" or "Test crashes on Y"
3. **Refine test**: LLM regenerates with feedback
4. Repeat (max iterations)

## 5. Key Insights

### 5.1 Execution Feedback Critical
Prior work: Generate → validate (maybe)
Echo: Generate → Execute → Analyze → Refine (mandatory loop)

**Execution provides**:
- Objective correctness signal
- Real failure modes (not just LLM hallucination)
- Ground truth for refinement

### 5.2 Dual-Validation Oracle
Using patched version as oracle is **elegant**:
- Avoids needing perfect issue description
- Any test that distinguishes buggy/patched works
- More robust than semantic matching

**Innovation from e-Otter++** extended and integrated into single-agent flow.

### 5.3 Graph Retrieval + Auto-Execution = Practical
Two innovations together make system **deployable**:
- Graph retrieval: Precise context, not fuzzy similarity
- Auto-execution: No manual setup, truly automated

**Prior systems** assumed test commands known - unrealistic.

## 6. Technical Depth

### Query Refinement Strategy

**When is context insufficient?**
```python
def check_context_adequate(context, issue):
    prompt = f"""
    Given issue: {issue}
    And retrieved context: {context}
    
    Is this context sufficient to write a reproduction test?
    Consider: Do we have the focal function? Test framework? Setup code?
    
    Answer YES/NO and explain what's missing.
    """
    response = llm(prompt)
    return "YES" in response

# Loop:
while not adequate:
    # Expand query: broader keywords, include imports, include tests
    context = graph.query(expanded_query)
    adequate = check_context_adequate(context, issue)
```

**Refinement operators**:
- Add more keywords from issue
- Include test files from same module
- Include similar bug fixes (from past PRs)
- Broaden file scope

### Command Detection Algorithm

```
1. Look for test configuration files:
   - pytest.ini, setup.cfg, pyproject.toml → pytest
   - tox.ini → check test commands
   - Makefile → look for test target

2. Scan test files for framework markers:
   - "import pytest" → pytest
   - "import unittest" → unittest
   - "from nose" → nose
   - Custom test decorators

3. Check CI configuration:
   - .github/workflows/*.yml → test command
   - .travis.yml, .circleci/config.yml

4. Fallback: Try common commands (pytest, python -m unittest, make test)

5. Validate: Dry run to check command works
```

### Fail-to-Pass Logic

```python
def validate_reproduction(test_code, buggy_repo, patched_repo):
    # Apply test to buggy
    result_buggy = run_test(test_code, buggy_repo)
    
    # Apply test to patched
    result_patched = run_test(test_code, patched_repo)
    
    # Criteria:
    # 1. Buggy: FAIL (exception, assertion failure, error)
    # 2. Patched: PASS (no failures)
    
    if result_buggy.status == PASS:
        return False, "Test passes on buggy code"
    
    if result_patched.status != PASS:
        return False, "Test doesn't pass on patched code"
    
    return True, "Valid reproduction"
```

- Auto-detects execution (no manual config)

- Iterative refinement loop

## 8. Strengths

1. **End-to-end automation**: Issue → reproduction test, no human intervention
2. **Execution integration**: Actually runs tests (novel)
3. **Graph-enhanced retrieval**: Better than pure LLM localization
4. **Objective validation**: Fail-to-pass via execution, not LLM judgment
5. **Cost-effective**: 1 test vs 10-50 candidates
6. **Open-source**: Code available, reproducible

---

## 9. Limitations

1. **Python-focused**: Test frameworks, execution commands Python-specific
   - Could extend to Java (Maven/Gradle), JS (npm test), but needs adaptation
   
2. **Graph construction cost**: Need code graph database
   - Not discussed how expensive to build
   
3. **Single test assumption**: Some bugs need multiple test cases
   - But reproduction typically single test sufficient
   
4. **Patch generation quality**: Generated patch may not be correct fix
   - Still works for distinction if it changes behavior
   - But could give false positives if patch doesn't actually fix
   
5. **Environment setup**: Complex projects need DB, env vars, services
   - Echo may fail to set up correctly
   - Not discussed how handled

## 14. Conclusion

Echo demonstrates that **graph-enhanced retrieval + automatic execution + dual-validation** enables high-quality issue reproduction test generation.

**Key formula**:
```
Precise context (graph) + Correct execution (auto-detect) + Objective validation (dual-version) = 66.28% success
```

This is the first system to combine all three effectively.

**Takeaway**: For test generation, **execution matters**. Can't rely on LLM judgment alone. Need to actually run tests and verify behavior. Graph KG provides context to generate plausible tests; execution provides ground truth.

**Open question**: Can this approach generalize to non-Python ecosystems? The auto-detection logic seems portable.
