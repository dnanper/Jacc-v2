## 1. Problem Statement

**Domain-specific code generation**: Software in specific industries (automotive, finance, healthcare) requires **specialized knowledge**:

### 1.1 LLM Knowledge Gap
- Most LLMs trained on public GitHub code
- Public code lacks **proprietary libraries**, internal APIs
- Domain-specific patterns underrepresented or absent
- Example from paper (Volvo truck software):
  ```python
  from utils.can import read  # Proprietary CAN bus library
  signal = read("OverspeedWarningStat")  # Domain-specific signal
  # Custom mapping logic
  ```

### 1.2 Two Knowledge Types Needed

**Conceptual Knowledge** (top-down):
- Domain libraries, APIs, their purposes
- Type definitions, signatures
- Architectural patterns
- Constraints, invariants

**Experiential Knowledge** (bottom-up):
- How to use APIs in practice
- Common patterns, idioms
- Edge cases, error handling
- Example: "When reading CAN signals, always check `None` return"

### 1.3 Challenge: Adapting Generic LLMs

**Options**:
1. **Fine-tuning**: Expensive, needs domain data, maintenance overhead
2. **RAG**: Simple retrieval, but how to ensure coverage?
3. **Combined**: Best of both, but how to integrate?

**Key question**: How to effectively retrieve and combine conceptual + experiential knowledge for domain tasks?

---

## 2. Proposed Solution: DomAgent

Autonomous coding agent with **DomRetriever** module that unifies:
- **Top-down**: Knowledge graph reasoning (conceptual)
- **Bottom-up**: Case-based reasoning (experiential)

### 2.1 System Architecture

```
┌──────────────────────────────────────────────────┐
│                     DomAgent                      │
│  ┌────────────────────────────────────────────┐  │
│  │         DomRetriever (Core)                │  │
│  │  ┌─────────────┐      ┌─────────────────┐ │  │
│  │  │ Top-Down    │      │ Bottom-Up       │ │  │
│  │  │ KG Reasoning│◄─────► Case Retrieval  │ │  │
│  │  └─────────────┘      └─────────────────┘ │  │
│  │           │                    │            │  │
│  │           └────────┬───────────┘            │  │
│  │                    ▼                          │  │
│  │         Integrated Knowledge                 │  │
│  └────────────────────┬─────────────────────────┘  │
│                       ▼                            │
│  ┌────────────────────────────────────────────┐   │
│  │   LLM (Code Generator)                     │   │
│  │   - Prompt with KG concepts               │   │
│  │   - Prompt with exemplar cases            │   │
│  │   - Synthesize domain-adapted code        │   │
│  └────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
```

**DomRetriever** can also operate independently (without full agent).

### 2.2 Knowledge Graph Construction (Top-Down)

**Domain KG** represents conceptual knowledge:

**Nodes**:
- **Packages/Modules**: Domain libraries (e.g., `utils.can`, `vehicle.sensors`)
- **Functions/Methods**: API endpoints
- **Classes/Types**: Domain data structures (CAN signals, vehicle states)
- **Concepts**: High-level domain terms (Overspeed, BrakeStatus)
- **Parameters**: With types, constraints
- **Descriptions**: LLM-generated or documentation-derived

**Edges**:
- `BELONGS_TO`: Function → Package
- `CALLS`: Function → Function
- `RETURNS`: Function → Type
- `ACCEPTS`: Function → Parameter Type
- `RELATED_TO`: Conceptual similarity
- `INHERITS`: Type relationships

**Source**:
- Library documentation
- Source code analysis (AST)
- LLM extraction/summarization
- Domain expert input (if available)

**Example** (from paper):
```
Package: utils.can
  Function: read(signal_name: str) -> CanValue
  Type: CanValue (enum: Valid, Invalid, Timeout)
  Related: OverspeedWarningStat (signal name)
```

### 2.3 Case Base Construction (Bottom-Up)

**Case base** = examples of domain code usage.

**Source**: Existing codebase (repositories using the domain library):
```python
# Case 1: Basic read
from utils.can import read
status = read("EngineTemperature")
if status.is_valid():
    process(status.value)

# Case 2: Error handling
try:
    signal = read("BrakePressure")
except CANError:
    log_error()
```

**Challenge**: Large case bases expensive to store/retrieve.

**DomAgent solution**: **KG-guided case selection** to maximize coverage with minimal cases.

---

### 2.4 KG-Guided Case Selection

**Goal**: Small case set (100-1000) that covers most domain concepts/APIs.

**Hierarchical approach**:

**Step 1: Package-level anchoring**
- For each package P in KG:
  - Select cases using functions from P
  - Ensure every package represented

**Step 2: Function-level clustering**
- Within package, cluster by function semantics:
  - Similar function names → same cluster
  - Embedding similarity of usage patterns
- Pick representative case from each cluster

**Step 3: Coverage validation**
- Check selected cases cover KG nodes:
  - What % of functions appear in cases?
  - What % of packages?
  - Aim: High coverage with minimal cases

**Result**: Compact, diverse case base (e.g., 500 cases covering 2000+ APIs)

---

### 2.5 Dual-Retrieval Integration

**During code generation**:

**Top-down retrieval** (KG):
```cypher
// Query: "Read CAN signal and map to standardized format"
MATCH (p:Package {name: 'utils.can'})
MATCH (p)-[:CONTAINS]->(f:Function)
WHERE f.name CONTAINS 'read' OR f.description CONTAINS 'signal'
RETURN f.name, f.signature, f.description
```

**Bottom-up retrieval** (cases):
- Embed query + case embeddings
- Find most similar cases
- Return full code examples

**Integration**:
- LLM receives both:
  - **Conceptual**: API signatures, types, descriptions
  - **Experiential**: Full working examples
- LLM synthesizes combining both knowledge sources

---

### 2.6 Reinforcement Learning for Retrieval

**Optional agent training** (mentioned in methodology):

**Policy**: Learn when to use KG vs cases, how to combine.

**Reward**:
- Generated code passes tests → +1
- Uses correct APIs → +0.5
- Follows domain patterns → +0.5

**Algorithm**: REINFORCE (policy gradient)

**Result**: Retrieval policy improves over time from feedback.

## 3. Evaluation

### Datasets
1. **DS-1000**: Data science domain benchmark (NumPy, Pandas, etc.)
   - 1000 tasks requiring domain-specific API knowledge
   
2. **Real-world truck software** (Volvo):
   - CAN bus signal reading
   - Vehicle state management
   - Diagnostic functions

### Baselines
- **Base LLM** (GPT-4o, Claude, etc.) without retrieval
- **Pure KG retrieval**: Only conceptual, no cases
- **Pure CBR**: Only cases, no KG structure
- **Simple RAG**: BM25/embedding retrieval from docs

### Results (DS-1000)

| Model | Accuracy |
|-------|----------|
| GPT-4o (baseline) | 45% |
| Pure KG | 52% |
| Pure CBR | 55% |
| Simple RAG | 58% |
| **DomAgent** | **67%** |

**Improvement**: +22% relative over baseline, +9% over simple RAG.

### Results (Truck Domain)

**Qualitative**: Successfully generated working code for:
- CAN signal reading with proper error handling
- Mapping proprietary formats to standards
- Integrating with existing truck software patterns

**Quantitative**: Not provided in excerpt, but "significantly enhances" claimed.

### Ablation Studies

| Component | DS-1000 Acc | Δ |
|-----------|-------------|---|
| Full DomAgent | 67% | - |
| - KG-guided case selection | 62% | -5% |
| - RL-trained retriever | 64% | -3% |
| - Dual-retrieval (only KG) | 52% | -15% |
| - Dual-retrieval (only cases) | 55% | -12% |

**Finding**: Both KG and cases needed; integration critical.

## 4. Key Insights

### 4.1 Top-Down + Bottom-Up Synergy
- **KG** provides structure: "These APIs exist, here's their signatures"
- **Cases** provide patterns: "This is how you typically use them together"
- Together: Complete picture

**Analogy**: Learning to drive
- **Conceptual**: Know what steering wheel, pedals do (KG)
- **Experiential**: See examples of merging, parking (cases)
- Need both

### 4.2 Coverage-Oriented Case Selection
Most CBR systems: Large case base, retrieve top-K
DomAgent: **Small curated set** via KG guidance

**Why better**:
- KG ensures diverse coverage (not redundant)
- Small set → fast retrieval, low storage
- High-quality representative examples

### 4.3 Domain Adaptation Without Fine-Tuning
Fine-tuning expensive, needs data, hard to maintain.
DomAgent: **Plug-and-play** knowledge injection
- Build KG + case base once
- Use with any LLM
- Update knowledge independently

**Enables small models**: Open-source models (7B-34B) achieve near-proprietary performance when augmented with domain KG.

## 5. Technical Implementation

### KG Construction for Domains

**Source integration**:
```python
# Parse library docs
docs = parse_sphinx_docs("utils.can")
for func in docs.functions:
    kg.add_node(func.name, type="Function", 
                signature=func.signature,
                description=func.docstring)
    kg.add_edge(func, package, "BELONGS_TO")

# LLM enhancement
for func in kg.functions:
    summary = llm(f"Summarize usage of {func.name}: {func.description}")
    func.summary = summary
```

**Embedding generation**:
- Use sentence transformer (MiniLM)
- Embed function names + descriptions
- Enable semantic search in KG

### Case Base from Existing Code

**Collection**:
```bash
# Search repositories using domain library
gh search code "from utils.can import" --language=python --limit=1000

# Extract usage snippets
for repo in repos:
    for file in repo.find_imports("utils.can"):
        cases.append(extract_usage_context(file))
```

**Clustering**:
```python
# Embed each case
embeddings = [encode(case.code) for case in cases]

# Cluster by API usage patterns
from sklearn.cluster import KMeans
clusters = KMeans(n_clusters=200).fit(embeddings)

# Select medoid from each cluster
for cluster in clusters:
    representative = find_medoid(cluster.embeddings)
    case_base.add(representative)
```

### Dual-Retrieval at Inference

```python
def retrieve(query, kg, case_base, k=5):
    # Top-down: KG query
    kg_results = kg.query(query, top_k=k)
    
    # Bottom-up: Case similarity
    query_emb = encode(query)
    case_similarities = [cosine(query_emb, case.emb) for case in case_base]
    top_cases = top_k(case_similarities, k=k)
    
    # Combine
    return {
        "kg_concepts": kg_results,
        "example_cases": top_cases
    }
```

**Integration into prompt**:
```
You are a domain expert in truck software.

Available APIs (from KG):
{formatted_kg_results}

Example usage patterns:
{formatted_cases}

Task: {query}

Generate code following domain conventions.
```

## 8. Limitations

1. **KG construction still manual**: Need domain schema, source docs
   - Could use CodeKGC approach to automate
   
2. **Case base curation**: Requires existing domain code
   - Cold start problem for brand-new domains
   
3. **Embedding quality**: Depends on good encoders
   - Domain-specific vocabulary may need fine-tuned encoder
   
4. **RL training overhead**: Requires reward signal (tests)
   - Not all domains have good test suites
   
5. **Scalability**: Large KGs (10K+ APIs) may slow retrieval
   - Need indexing, caching

## 11. Significance

DomAgent addresses **practical domain adaptation**:
- Companies have proprietary libraries
- Cannot fine-tune on all of them
- Need plug-and-play knowledge injection

**Result**: Small open-source models achieve near-proprietary LLM performance on domain tasks.

**Pattern**: 
1. Build domain KG (concepts, APIs)
2. Curate small case base (usage patterns)
3. Use KG to ensure coverage
4. Retrieve both, feed to LLM

This pattern generalizes beyond domains to any specialized knowledge work.

## 14. Conclusion

DomAgent demonstrates that **structured conceptual knowledge (KG) + experiential patterns (CBR)** enables effective domain-specific code generation without model fine-tuning.

**Key formula**:
```
Domain Knowledge = Concepts (KG) + Patterns (Cases)
Code Generation = LLM + Dual Retrieval
```

**Results**: 67% on DS-1000 (competitive with fine-tuned models), successful industrial deployment.

**Takeaway**: For specialized domains, invest in **knowledge representation** (KG + cases) rather than model customization. This is more maintainable, portable, and cost-effective.

**Broader implication**: This "knowledge-first" approach may extend beyond domains to any scenario where LLMs lack specific knowledge. Instead of retraining, **externalize knowledge** in structured form and retrieve intelligently.

DomAgent represents a **pragmatic middle path** between naive RAG and expensive fine-tuning. The KG-guided case selection is particularly clever - ensures quality coverage without massive case libraries.
