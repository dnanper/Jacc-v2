## 1. Tuyên bố vấn đề

**Thử nghiệm fuzz** tính hiệu quả phụ thuộc vào **trình điều khiển fuzz** - mã thực hiện các API mục tiêu với đầu vào được tạo. Vấn đề:

### 1.1 Viết trình điều khiển thủ công
- Tốn thời gian, đòi hỏi kiến ​​thức chuyên môn
- Nhà phát triển có thể không biết cách kết hợp API tốt nhất
- Giới hạn phạm vi kiểm tra

### 1.2 Những thách thức về tạo trình điều khiển dựa trên LLM
- **Lỗi biên dịch**: LLM tạo mã không hợp lệ về mặt cú pháp
- **Lạm dụng API**: Sai thứ tự, thiếu khởi tạo, loại không chính xác
- **Không hiểu về cách sử dụng**: Cách kết hợp các API thường
- **Tạo hạt giống**: Cần đầu vào ban đầu hợp lệ cho đột biến
- **Phân tích sự cố**: Phân biệt lỗi trình điều khiển với lỗi thư viện

### 1.3 Bộ làm mờ LLM hiện tại
- NhắcFuzz: Kết hợp đột biến API nhưng không hiểu ngữ cảnh
- OSS-Fuzz-Gen: Không/vài lần bắn, nhưng mù quáng về các chi tiết cụ thể của cơ sở mã
- Thiếu: Kiến thức cụ thể về kho lưu trữ, cách sử dụng, luồng dữ liệu

**Mục tiêu**: Tạo trình điều khiển fuzz hoàn toàn tự động với **LLM nhận biết cơ sở mã**.

### 2.2 Xây dựng sơ đồ tri thức

**Phân tích chương trình liên thủ tục**:

**Nút**:
- Hàm (có chữ ký, kiểu trả về, tham số)
- Cấu trúc dữ liệu (cấu trúc, lớp)
- Tệp/mô-đun
- Biến toàn cục

**Cạnh**:
- **Cuộc gọi**: Chức năng A gọi Chức năng B
- **Định nghĩa**: Tệp/lớp chứa hàm
- **Công dụng**: Hàm sử dụng biến/loại
- **Đọc/Ghi**: Các kiểu truy cập bộ nhớ
- **Loại mối quan hệ**: Kế thừa, thành phần

**Siêu dữ liệu bổ sung** (do LLM tạo):
- Tóm tắt chức năng (nó làm gì)
- Kịch bản sử dụng (khi nào nên sử dụng)
- Ngữ nghĩa tham số (phạm vi hợp lệ, ràng buộc)
- Đoạn mã ví dụ

**Bộ nhớ**: Cơ sở dữ liệu đồ thị (Neo4j được đề cập hoặc tương tự)

### 2.3 Tác nhân kết hợp API

**Nhiệm vụ**: Xác định API nào sẽ kết hợp trong trình điều khiển fuzz.

**Cách tiếp cận truyền thống**: Thử tất cả các kết hợp → bùng nổ tổ hợp.

**Phương pháp tiếp cận CKGFuzzer**:

1. **Truy vấn KG về sự xuất hiện cùng lúc**:
``` cypher
MATCH (f1:Function)-[:CALLED_WITHIN]->(ctx)←[:CALLED_WITHIN]-(f2)
Ở đâu f1 ≠ f2
RETURN f1, f2, count(ctx) là sự xuất hiện
ĐẶT HÀNG THEO sự xuất hiện đồng thời DESC
   ```

2. **Tìm các kiểu sử dụng điển hình**:
   - Ban đầu → Cấu hình → Thực thi → Các mẫu dọn dẹp
   - Các mẫu xử lý lỗi (kiểm tra giá trị trả về)
   - Quản lý tài nguyên (cấp phát, sử dụng, miễn phí)

3. **Ưu tiên**:
   - Tỷ lệ xuất hiện cao = khả năng kết hợp hợp lệ
   - Bao gồm các loại dữ liệu khác nhau
   - Bao gồm các trường hợp cạnh (đầu vào NULL, giá trị biên)

### 2.4 Tác nhân tạo trình điều khiển Fuzz

**Thế hệ dựa trên LLM với bối cảnh KG**:

**Lời nhắc bao gồm**:
- API mục tiêu (chữ ký từ KG)
- Các kiểu sử dụng điển hình (từ KG)
- Các chức năng liên quan (để thiết lập/phân tách)
- Định nghĩa kiểu (structs, enums)
- Các mẫu xử lý lỗi

**Cấu trúc trình điều khiển được tạo**:
```c
#include "target_lib.h"

// Global state
static lib_context_t ctx;

// Input structure
typedef struct {
    uint8_t* data;
    size_t len;
} input_t;

// Fuzz driver entry point
int LLVMFuzzerTestOneInput(const uint8_t* data, size_t len) {
    // 1. Initialize (from KG: required setup)
    lib_init(&ctx);
    
    // 2. Parse input (KG: expected format)
    input_t inp = parse_input(data, len);
    
    // 3. Call target API (from combination agent)
    result_t res = lib_process(&ctx, inp.param1, inp.param2);
    
    // 4. Check result (KG: error codes to handle)
    if (res.status != SUCCESS) {
        lib_cleanup(&ctx);
        return 0;
    }
    
    // 5. Additional API calls (from combination)
    lib_validate(&ctx, res.output);
    
    // 6. Cleanup
    lib_cleanup(&ctx);
    return 0;
}
```

**KG cung cấp**:
- Những chức năng init/dọn dẹp nào kết hợp với mục tiêu
- Các loại tham số, phạm vi hợp lệ
- Mã lỗi cần kiểm tra
- Yêu cầu an toàn chủ đề

### 2.5 Tác nhân sửa chữa biên dịch

**Sự cố**: Mã do LLM tạo có thể không biên dịch được.

**Quy trình sửa chữa**:

1. **Cố gắng biên soạn**: `clang -c driver.c -o driver.o`

2. **Lỗi phân tích cú pháp**:
   - Thiếu bao gồm
   - Loại không khớp
   - Ký hiệu không xác định
   - Chữ ký sai

3. **Truy vấn KG để sửa lỗi**:
``` cypher
// Thiếu hàm: tìm chữ ký đúng
MATCH (f:Hàm {name: 'lib_init'})
TRẢ LẠI f.signature, f.header_file
   
// Loại không xác định: tìm định nghĩa
MATCH (t:Gõ {name: 'lib_context_t'})
TRẢ LẠI t.định nghĩa, t.module
   ```

4. **Sửa chữa LLM**: Cung cấp lỗi + thông tin KG → tái tạo/sửa chữa

5. **Lặp lại** (tối đa N lần thử)

**Kết quả**: Trình điều khiển được biên dịch hợp lệ

### 2.6 Đột biến theo hướng dẫn bảo hiểm

**Làm mờ tiêu chuẩn**: Thay đổi đầu vào, theo dõi phạm vi mã, ưu tiên phạm vi mới.

**Cải tiến CKGFuzzer**:

**Đột biến kết hợp API**:
- Nếu sự kết hợp hiện tại không tăng phạm vi bảo hiểm:
  - API thay thế (từ KG)
  - Sắp xếp lại lệnh gọi API (từ mẫu thời gian của KG)
  - Thêm thiết lập/dọn dẹp bị thiếu (từ KG)

**Tạo hạt giống từ luồng dữ liệu**:
- Phân tích luồng dữ liệu của trình điều khiển (từ KG)
- Tạo hạt giống thực hiện các con đường khác nhau
- Ví dụ: nếu trình điều khiển phân tích cú pháp trường đầu vào `type`, hãy tạo hạt giống cho từng giá trị enum

**Phản hồi cho Đại lý kết hợp**:
- Những combo API nào đạt được mức độ bao phủ cao?
- Cập nhật mức độ ưu tiên kết hợp

## 4. Những hiểu biết chính

### 4,1 KG cho các kiểu sử dụng API
Không chỉ có chữ ký API mà còn **cách các API thường được sử dụng cùng nhau**:
- Khởi tạo → vận hành → trình tự dọn dẹp
- Lỗi xử lý thành ngữ
- Các mẫu an toàn của chủ đề
- Mẫu cấu hình

Kiến thức về "cách sử dụng thông thường" này là **theo kinh nghiệm** (từ mã hiện có), không chỉ từ tài liệu.

### 4.2 Chuyên môn hóa đa tác nhân
- **Tác nhân kết hợp**: Biết nên gọi API nào
- **Generation Agent**: Biết viết code
- **Repair Agent**: Biết cách sửa lỗi biên dịch
- **Fuzz Engine**: Biết cách kiểm tra

Mỗi chuyên ngành → tốt hơn LLM nguyên khối.

### 4.3 Tinh chỉnh lặp đi lặp lại
- Không phải thế hệ một lần
- Biên dịch → vòng lặp sửa chữa
- Phản hồi vùng phủ sóng → đột biến kết hợp
- Phân tích sự cố → cải tiến trình điều khiển

**Khóa**: KG được cập nhật các phát hiện (cơ sở kiến ​​thức phát triển trong quá trình làm mờ).

## 5. Chiều sâu kỹ thuật

### Phân tích chương trình cho KG

**Phân tích liên thủ tục**:
- Xây dựng biểu đồ cuộc gọi trên các chức năng
- Theo dõi luồng dữ liệu (chuỗi khử sử dụng)
- Xác định các điểm vào (hàm mục tiêu mờ)
- Khám phá các chức năng trợ giúp/thiết lập

**Công cụ**:
- LLVM/Clang cho C/C++
- Người trông cây cho nhiều ngôn ngữ
- Phân tích tùy chỉnh cho các mẫu dành riêng cho thư viện

### Chi tiết đột biến theo hướng dẫn bảo hiểm

**Tiêu chuẩn**: Thuật toán di truyền của AFL
- Thay đổi byte/số float/số nguyên
- Giữ hạt tăng độ che phủ cạnh

**CKGFuzzer bổ sung**:
- **Đột biến cấp độ API**: Hoán đổi lệnh gọi API này sang lệnh gọi API khác
- **Đột biến tham số**: Dựa trên thông tin loại từ KG
- **Đột biến luồng điều khiển**: Thêm/xóa kiểm tra lỗi (từ mẫu KG)

**Ví dụ**:
```
Bản gốc: lib_init(&ctx); lib_process(&ctx, data, len);
Đã đột biến: lib_init(&ctx); lib_configure(&ctx, timeout=1000); lib_process(&ctx, data, len);
```

## 9. Chi tiết kỹ thuật: Ví dụ truy vấn KG

**Đối với kết hợp API**:
```cypher
// Find APIs commonly called together
MATCH (f1:Function)-[:CALLED_IN]->(ctx:Context)-[:CALLED_IN]->(f2)
WHERE f1.name = 'lib_init' AND f2.name <> 'lib_init'
RETURN f2.name, count(*) as frequency
ORDER BY frequency DESC
LIMIT 10
```

**Để sửa chữa biên dịch**:
```cypher
// Get function signature
MATCH (f:Function {name: $func_name})
RETURN f.signature, f.defined_in, f.requires

// Get type definition
MATCH (t:Type {name: $type_name})-[:DEFINED_IN]->(f:File)
RETURN f.path, t.definition
```

**Để phân tích sự cố**:
```cypher
// Find similar crash contexts
MATCH (f:Function)-[:CALLS]->(crashed_func)
WHERE crashed_func.has_bug = true
RETURN f, pattern of calls
```
