## 1. Tuyên bố vấn đề

**Tạo thử nghiệm tái tạo sự cố**: Đưa ra báo cáo lỗi (sự cố GitHub), tự động tạo trường hợp thử nghiệm tái tạo lỗi.

**Tại sao điều này lại quan trọng**:
- Các nhà phát triển dành rất nhiều thời gian để tìm hiểu và tái tạo lỗi
- Kiểm tra tái tạo xác nhận tính chính xác của bản sửa lỗi (thất bại trước, vượt qua sau)
- Cần thiết cho cổng chất lượng CI/CD
- Phân loại lỗi yêu cầu sao chép

**Thử thách**:

### 1.1 Mô tả vấn đề mơ hồ
- Báo cáo vấn đề thường không đầy đủ
- Thiếu mục nhập, mã thiết lập, hành vi dự kiến
- Có thể bao gồm một phần đoạn mã không có ngữ cảnh
- Ví dụ từ giấy (Django-12284):
  ```
"Không thể ghi đè get_FOO_display() trong Django 2.2+"
+ đoạn mã tối thiểu, không nhập, không khung kiểm tra
  ```

### 1.2 Kiểm tra kho lưu trữ cụ thể
- Mỗi dự án sử dụng khung kiểm tra khác nhau (pytest, unittest, custom)
- Các quy ước khác nhau về đặt tên bài kiểm tra, vị trí, cấu trúc
- Các lệnh thực thi cụ thể (pytest django/test/...)
- Thiết lập môi trường (cơ sở dữ liệu, cài đặt, vars env)

### 1.3 Thiếu Oracle đáng tin cậy
- Làm thế nào để xác minh thử nghiệm được tạo là chính xác?
- Mô tả vấn đề có thể không nêu chính xác lỗi dự kiến
- Đánh giá ngữ nghĩa dựa trên LLM không đáng tin cậy
- Cần tiêu chí không vượt qua: kiểm tra không thành công khi có lỗi, vượt qua đã sửa

### 1.4 Thiết lập thực thi
- Chạy thử nghiệm yêu cầu đúng lệnh, môi trường
- Các vấn đề không cung cấp thông tin này
- Không khớp nhỏ → kiểm tra không chạy → không thể xác thực

## 2. Giải pháp đề xuất: Echo

Hệ thống đại lý với 3 điểm đổi mới quan trọng:

### 2.1 Truy xuất bối cảnh nâng cao bằng đồ thị

**Vấn đề**: Truy xuất truyền thống (bản địa hóa tệp LLM phân cấp của Agentless) thiếu hiểu biết về cấu trúc.

**Giải pháp**: Sử dụng **biểu đồ mã** để biết ngữ cảnh chính xác.

**Tính năng đồ thị**:
- Thực thể mã (tệp, lớp, hàm)
- Mối quan hệ cuộc gọi
- Nhập phần phụ thuộc
- Vị trí và mẫu thử nghiệm

**Tinh chỉnh truy vấn tự động**:
- Truy vấn ban đầu từ mô tả vấn đề
- Kiểm tra xem ngữ cảnh được truy xuất có đủ không (đánh giá LLM)
- Nếu không đủ → tinh chỉnh truy vấn, truy xuất thêm
- Lặp lại cho đến ngưỡng đầy đủ ngữ cảnh

**Bộ ngữ cảnh nhỏ gọn**:
- Không phải tất cả thông tin được truy xuất đều được gửi đến trình tạo
- Lọc để phù hợp nhất
- Giảm tiếng ồn, cải thiện chất lượng thế hệ

### 2.2 Thực hiện kiểm thử tự động

**Tính năng mới**: Hệ thống đầu tiên tích hợp liền mạch việc thực hiện kiểm thử vào quá trình tạo bản sao.

**Quy trình làm việc**:
1. Tạo bài kiểm tra ứng viên
2. **Tự động xác định lệnh thực thi**:
   - Phân tích cấu trúc kho lưu trữ
   - Phát hiện khung kiểm tra (pytest vs unittest vs custom)
   - Tìm cấu hình người chạy thử
   - Xây dựng lệnh phù hợp
3. **Thực hiện kiểm thử** trong môi trường biệt lập
4. **Ghi lại kết quả**: Đạt/không đạt, thông báo lỗi, dấu vết ngăn xếp

**Tại sao quan trọng**:
- Xác thực thử nghiệm thực sự chạy
- Cung cấp phản hồi thực hiện để sàng lọc
- Cho phép xác thực không vượt qua được

### 2.3 Xác thực phiên bản kép bằng cách tạo bản vá

**Thông tin chi tiết quan trọng** (từ e-Otter++): Sử dụng phiên bản vá lỗi làm oracle.

**Quá trình**:
1. Tạo bài kiểm tra ứng viên T
2. Tạo bản vá P tiềm năng (sẽ sửa lỗi)
3. Áp dụng P cho codebase → phiên bản vá
4. Chạy T trên:
   - **Cơ sở mã gốc**: Sẽ THẤT BẠI
   - **Cơ sở mã đã được vá**: Nên ĐẠT
5. Nếu T thể hiện hành vi không vượt qua → sao chép hợp lệ
6. Nếu không, hãy sử dụng phân tích lỗi để tinh chỉnh T

**Lợi thế so với phán đoán LLM thuần túy**:
- Tiêu chí dựa trên việc thực hiện khách quan
- Không cần LLM để khớp các lỗi về mặt ngữ nghĩa
- Xác thực đáng tin cậy hơn

### 2.4 Trọng tâm Kiểm tra Đơn

**Không giống như các phương pháp trước**:
- LIBRO, AEGIS, e-Otter++: Tạo nhiều ứng viên, chọn người giỏi nhất
- Echo: Tạo **một bài kiểm tra chất lượng cao** thông qua sàng lọc lặp đi lặp lại

**Lý do**:
- Tiết kiệm chi phí hơn (1 bài kiểm tra so với 10-50 thí sinh)
- Đánh đổi hiệu suất chi phí tốt hơn
- Đủ cho mục đích sinh sản

## 3. Triển khai kỹ thuật

### 3.1 Xây dựng đồ thị mã

**Nguồn**:
- Phân tích AST tĩnh (chức năng, lớp, lệnh gọi)
- Kiểm tra nhận dạng tập tin
- Phát hiện khung (đánh dấu pytest, lớp unittest)
- Mối quan hệ nhập khẩu

**Truy vấn đồ thị để truy xuất**:
```cypher
// Find relevant functions
MATCH (f:Function)-[:CONTAINS_IN]->(file)
WHERE f.name CONTAINS $keyword OR f.docstring CONTAINS $keyword
RETURN f

// Find test files and patterns
MATCH (t:TestFile)-[:USES_FRAMEWORK]->(fw)
RETURN t.path, fw.name, t.structure_pattern
```

### 3.2 Tự động phát hiện lệnh

**Heuristic**:
- `pytest.ini`, `setup.cfg`, `pyproject.toml` → pytest
- `unittest` nhập trong thử nghiệm → không đáng tin cậy
- Tập lệnh chạy tùy chỉnh → phân tích cú pháp cho mẫu lệnh gọi

**Cấu trúc lệnh**:
```python
if framework == "pytest":
    if "django" in repo:
        cmd = "pytest --ds=test_settings"
    else:
        cmd = "pytest"
elif framework == "unittest":
    cmd = "python -m unittest discover"
```

### 3.3 Tạo bản vá để xác thực

**Cần**: Tạo bản vá hợp lý có thể khắc phục sự cố.

**Tiếp cận**:
- LLM tạo bản sửa lỗi ứng viên đã cho + mã tiêu điểm
- Không cần phải hoàn hảo (chỉ cần thay đổi hành vi)
- Các sửa chữa đơn giản thường là đủ: quay lại sớm, báo lỗi, sửa đổi điều kiện

**Tại sao tính năng này hoạt động**:
- Ngay cả bản vá không hoàn hảo cũng có thể thay đổi kết quả kiểm tra
- Kiểm tra không thành công trên cả lỗi và bản vá → không hợp lệ
- Kiểm tra chỉ thất bại khi có lỗi → sao chép hợp lệ

### 3.4 Vòng phân tích lỗi

Nếu kiểm tra không đạt, hãy kiểm tra:
1. **Phân tích lỗi**:
   - Kiểm tra lỗi trước khi vá? → bản thân bài kiểm tra bị hỏng
   - Kiểm tra vượt qua cả hai? → không tạo ra lỗi
   - Kiểm tra thất bại cả hai? → bản vá quá rộng
2. **Tạo phản hồi**: "Kiểm tra cần kiểm tra X" hoặc "Kiểm tra sự cố trên Y"
3. **Tinh chỉnh thử nghiệm**: LLM tái tạo bằng phản hồi
4. Lặp lại (số lần lặp tối đa)

## 5. Những hiểu biết chính

### 5.1 Phản hồi thực thi Quan trọng
Công việc trước đây: Tạo → xác thực (có thể)
Echo: Tạo → Thực thi → Phân tích → Tinh chỉnh (vòng lặp bắt buộc)

**Việc thực thi cung cấp**:
- Tín hiệu chính xác khách quan
- Các chế độ thất bại thực sự (không chỉ ảo giác LLM)
- Sự thật căn bản để sàng lọc

### 5.2 Oracle xác thực kép
Sử dụng phiên bản vá lỗi như oracle là **thanh lịch**:
- Tránh cần mô tả vấn đề hoàn hảo
- Bất kỳ thử nghiệm nào để phân biệt các tác phẩm có lỗi/được vá
- Mạnh mẽ hơn so với kết hợp ngữ nghĩa

**Sự đổi mới từ e-Otter++** được mở rộng và tích hợp vào luồng tác nhân đơn.

### 5.3 Truy xuất đồ thị + Tự động thực thi = Thực tế
Hai cải tiến cùng nhau làm cho hệ thống **có thể triển khai**:
- Truy xuất đồ thị: Ngữ cảnh chính xác, không có sự tương đồng mờ nhạt
- Tự động thực thi: Không cần thiết lập thủ công, thực sự tự động

**Các hệ thống trước** giả định các lệnh kiểm tra đã biết - không thực tế.

## 6. Chiều sâu kỹ thuật

### Chiến lược sàng lọc truy vấn

**Khi nào ngữ cảnh không đủ?**
```python
def check_context_adequate(context, issue):
    prompt = f"""
    Given issue: {issue}
    And retrieved context: {context}
    
    Is this context sufficient to write a reproduction test?
    Consider: Do we have the focal function? Test framework? Setup code?
    
    Answer YES/NO and explain what's missing.
    """
    response = llm(prompt)
    return "YES" in response

# Loop:
while not adequate:
    # Expand query: broader keywords, include imports, include tests
    context = graph.query(expanded_query)
    adequate = check_context_adequate(context, issue)
```

**Toán tử sàng lọc**:
- Thêm từ khóa khác từ vấn đề
- Bao gồm các tệp thử nghiệm từ cùng một mô-đun
- Bao gồm các bản sửa lỗi tương tự (từ các PR trước đây)
- Mở rộng phạm vi tập tin

### Thuật toán phát hiện lệnh

```
1. Tìm kiếm các tập tin cấu hình thử nghiệm:
   - pytest.ini, setup.cfg, pyproject.toml → pytest
   - tox.ini → kiểm tra các lệnh kiểm tra
   - Makefile → tìm mục tiêu thử nghiệm

2. Quét các tệp kiểm tra để tìm điểm đánh dấu khung:
   - "nhập pytest" → pytest
   - "nhập khẩu unittest" → unittest
   - “từ mũi” → mũi
   - Trang trí thử nghiệm tùy chỉnh

3. Kiểm tra cấu hình CI:
   - .github/workflows/*.yml → lệnh kiểm tra
   - .travis.yml, .circleci/config.yml

4. Dự phòng: Thử các lệnh phổ biến (pytest, python -m unittest, make test)

5. Xác thực: Chạy khô để kiểm tra lệnh hoạt động
```

### Logic không vượt qua được

```python
def validate_reproduction(test_code, buggy_repo, patched_repo):
    # Apply test to buggy
    result_buggy = run_test(test_code, buggy_repo)
    
    # Apply test to patched
    result_patched = run_test(test_code, patched_repo)
    
    # Criteria:
    # 1. Buggy: FAIL (exception, assertion failure, error)
    # 2. Patched: PASS (no failures)
    
    if result_buggy.status == PASS:
        return False, "Test passes on buggy code"
    
    if result_patched.status != PASS:
        return False, "Test doesn't pass on patched code"
    
    return True, "Valid reproduction"
```

4. **Chất lượng tạo bản vá**: Bản vá được tạo có thể không phải là bản sửa lỗi chính xác

- Mục tiêu, dựa trên thực hiện

### 4. Tinh chỉnh lặp lại thử nghiệm đơn
- Tập trung chất lượng hơn số lượng

## 14. Kết luận

Echo chứng minh rằng **truy xuất nâng cao bằng đồ thị + thực thi tự động + xác thực kép** cho phép tạo thử nghiệm tái tạo vấn đề chất lượng cao.

**Công thức chính**:
```
Ngữ cảnh chính xác (biểu đồ) + Thực thi chính xác (tự động phát hiện) + Xác thực khách quan (phiên bản kép) = thành công 66,28%
```

Đây là hệ thống đầu tiên kết hợp cả ba một cách hiệu quả.

**Bài học rút ra**: Đối với việc tạo thử nghiệm, **vấn đề thực thi**. Không thể chỉ dựa vào phán đoán của LLM. Cần thực sự chạy thử nghiệm và xác minh hành vi. Graph KG cung cấp bối cảnh để tạo ra các thử nghiệm hợp lý; việc thực thi cung cấp sự thật cơ bản.

**Câu hỏi mở**: Cách tiếp cận này có thể khái quát hóa cho các hệ sinh thái không phải Python không? Logic tự động phát hiện có vẻ di động.
