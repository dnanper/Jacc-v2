## 1. Tuyên bố vấn đề

**Tạo mã dành riêng cho miền**: Phần mềm trong các ngành cụ thể (ô tô, tài chính, chăm sóc sức khỏe) yêu cầu **kiến thức chuyên môn**:

### 1.1 Khoảng cách kiến ​​thức LLM
- Hầu hết các LLM được đào tạo về mã GitHub công khai
- Mã công khai thiếu **thư viện độc quyền**, API nội bộ
- Các mẫu dành riêng cho miền không được trình bày đầy đủ hoặc vắng mặt
- Ví dụ từ giấy (phần mềm xe tải Volvo):
``` con trăn
từ utils.can nhập đọc # Thư viện bus CAN độc quyền
signal = read("OverspeedWarningStat") # Tín hiệu dành riêng cho miền
  # Logic ánh xạ tùy chỉnh
  ```

### 1.2 Cần có hai loại kiến ​​thức

**Kiến thức khái niệm** (từ trên xuống):
- Thư viện miền, API, mục đích của chúng
- Định nghĩa kiểu, chữ ký
- Mẫu kiến ​​trúc
- Những ràng buộc, bất biến

**Kiến thức kinh nghiệm** (từ dưới lên):
- Cách sử dụng API trong thực tế
- Những mẫu câu, thành ngữ thông dụng
- Các trường hợp cạnh, xử lý lỗi
- Ví dụ: "Khi đọc tín hiệu CAN, luôn kiểm tra `None` return"

### 1.3 Thách thức: Điều chỉnh LLM chung

**Tùy chọn**:
1. **Tinh chỉnh**: Đắt tiền, cần dữ liệu miền, chi phí bảo trì
2. **RAG**: Truy xuất đơn giản nhưng làm thế nào để đảm bảo phạm vi phủ sóng?
3. **Kết hợp**: Tốt nhất của cả hai, nhưng làm thế nào để tích hợp?

**Câu hỏi chính**: Làm thế nào để truy xuất và kết hợp hiệu quả kiến ​​thức khái niệm + kinh nghiệm cho các nhiệm vụ trong miền?

### 2.2 Xây dựng sơ đồ tri thức (Từ trên xuống)

**Miền KG** thể hiện kiến ​​thức khái niệm:

**Nút**:
- **Gói/Mô-đun**: Thư viện miền (ví dụ: `utils.can`, `vehicle.sensors`)
- **Hàm/Phương thức**: Điểm cuối API
- **Lớp/Loại**: Cấu trúc dữ liệu miền (tín hiệu CAN, trạng thái phương tiện)
- **Khái niệm**: Thuật ngữ miền cấp cao (Overspeed, BrakeStatus)
- **Thông số**: Với các loại, ràng buộc
- **Mô tả**: Do LLM tạo hoặc bắt nguồn từ tài liệu

**Cạnh**:
- `BELONGS_TO`: Chức năng → Gói
- `CALLS`: Hàm → Hàm
- `RETURNS`: Hàm → Loại
- `ACCEPTS`: Hàm → Loại tham số
- `RELATED_TO`: Sự tương đồng về khái niệm
- `INHERITS`: Nhập mối quan hệ

**Nguồn**:
- Tài liệu thư viện
- Phân tích mã nguồn (AST)
- Trích xuất/tóm tắt LLM
- Đầu vào của chuyên gia tên miền (nếu có)

**Ví dụ** (từ giấy):
```
Gói: utils.can
Hàm: read(signal_name: str) -> CanValue
Loại: CanValue (enum: Hợp lệ, Không hợp lệ, Hết giờ)
Liên quan: OverspeedWarningStat (tên tín hiệu)
```

### 2.3 Cấu trúc cơ sở trường hợp (Từ dưới lên)

**Cơ sở trường hợp** = ví dụ về cách sử dụng mã miền.

**Nguồn**: Codebase hiện có (kho sử dụng thư viện miền):
```python
# Case 1: Basic read
from utils.can import read
status = read("EngineTemperature")
if status.is_valid():
    process(status.value)

# Case 2: Error handling
try:
    signal = read("BrakePressure")
except CANError:
    log_error()
```

**Thách thức**: Đế hộp lớn đắt tiền để lưu trữ/truy xuất.

**Giải pháp DomAgent**: **Lựa chọn trường hợp theo hướng dẫn của KG** để tối đa hóa phạm vi bao phủ với số trường hợp tối thiểu.

---

### Lựa chọn trường hợp có hướng dẫn 2.4 KG

**Mục tiêu**: Tập hợp trường hợp nhỏ (100-1000) bao gồm hầu hết các khái niệm/API miền.

**Phương pháp phân cấp**:

**Bước 1: Neo cấp gói**
- Đối với mỗi gói P tính bằng KG:
  - Chọn trường hợp sử dụng hàm từ P
  - Đảm bảo mỗi gói được đại diện

**Bước 2: Phân cụm cấp độ chức năng**
- Trong gói, phân cụm theo ngữ nghĩa hàm:
  - Tên hàm tương tự → cùng cụm
  - Nhúng sự tương đồng của các mẫu sử dụng
- Chọn trường hợp đại diện từ mỗi cụm

**Bước 3: Xác thực phạm vi bảo hiểm**
- Kiểm tra các trường hợp đã chọn bao gồm các nút KG:
  - Bao nhiêu % chức năng xuất hiện trong các trường hợp?
  - Bao nhiêu % gói?
  - Mục tiêu: Độ bao phủ cao với số trường hợp tối thiểu

**Kết quả**: Cơ sở trường hợp nhỏ gọn, đa dạng (ví dụ: 500 trường hợp bao gồm hơn 2000 API)

### 2.5 Tích hợp truy xuất kép

**Trong quá trình tạo mã**:

**Truy xuất từ ​​trên xuống** (KG):
```cypher
// Query: "Read CAN signal and map to standardized format"
MATCH (p:Package {name: 'utils.can'})
MATCH (p)-[:CONTAINS]->(f:Function)
WHERE f.name CONTAINS 'read' OR f.description CONTAINS 'signal'
RETURN f.name, f.signature, f.description
```

**Truy xuất từ ​​dưới lên** (trường hợp):
- Nhúng truy vấn + nhúng trường hợp
- Tìm hầu hết các trường hợp tương tự
- Trả về ví dụ mã đầy đủ

**Tích hợp**:
- LLM nhận được cả hai:
  - **Khái niệm**: Chữ ký, loại, mô tả API
  - **Trải nghiệm**: Ví dụ hoạt động đầy đủ
- LLM tổng hợp kết hợp cả hai nguồn kiến ​​thức

### 2.6 Học tăng cường để truy hồi

**Đào tạo đại lý tùy chọn** (được đề cập trong phương pháp luận):

**Chính sách**: Tìm hiểu khi nào nên sử dụng KG với các trường hợp, cách kết hợp.

**Phần thưởng**:
- Mã được tạo đã vượt qua các bài kiểm tra → +1
- Sử dụng API chính xác → +0,5
- Theo mẫu miền → +0,5

**Thuật toán**: REINFORCE (độ dốc chính sách)

**Kết quả**: Chính sách truy xuất được cải thiện theo thời gian nhờ phản hồi.

## 5. Triển khai kỹ thuật

### KG Construction cho miền

**Tích hợp nguồn**:
```python
# Parse library docs
docs = parse_sphinx_docs("utils.can")
for func in docs.functions:
    kg.add_node(func.name, type="Function", 
                signature=func.signature,
                description=func.docstring)
    kg.add_edge(func, package, "BELONGS_TO")

# LLM enhancement
for func in kg.functions:
    summary = llm(f"Summarize usage of {func.name}: {func.description}")
    func.summary = summary
```

**Thế hệ nhúng**:
- Sử dụng biến đổi câu (MiniLM)
- Nhúng tên hàm + mô tả
- Bật tìm kiếm ngữ nghĩa trong KG

### Cơ sở trường hợp từ mã hiện có

**Bộ sưu tập**:
```bash
# Search repositories using domain library
gh search code "from utils.can import" --language=python --limit=1000

# Extract usage snippets
for repo in repos:
    for file in repo.find_imports("utils.can"):
        cases.append(extract_usage_context(file))
```

**Phân cụm**:
```python
# Embed each case
embeddings = [encode(case.code) for case in cases]

# Cluster by API usage patterns
from sklearn.cluster import KMeans
clusters = KMeans(n_clusters=200).fit(embeddings)

# Select medoid from each cluster
for cluster in clusters:
    representative = find_medoid(cluster.embeddings)
    case_base.add(representative)
```

### Truy xuất kép khi suy luận

```python
def retrieve(query, kg, case_base, k=5):
    # Top-down: KG query
    kg_results = kg.query(query, top_k=k)
    
    # Bottom-up: Case similarity
    query_emb = encode(query)
    case_similarities = [cosine(query_emb, case.emb) for case in case_base]
    top_cases = top_k(case_similarities, k=k)
    
    # Combine
    return {
        "kg_concepts": kg_results,
        "example_cases": top_cases
    }
```

**Tích hợp vào dấu nhắc**:
```
Bạn là chuyên gia về lĩnh vực phần mềm xe tải.

API có sẵn (từ KG):
{formatted_kg_results}

Mẫu sử dụng ví dụ:
{trường hợp được định dạng}

Nhiệm vụ: {truy vấn}

Tạo mã theo quy ước tên miền.
```

|**CKGFuzzer**|Kiến thức miền|DomAgent: thế hệ chung, không làm mờ|

**Duy nhất**: Lần đầu tiên kết hợp một cách có hệ thống **KG concept** + **CBR experience** cho gen mã miền.

## 11. Ý nghĩa

Địa chỉ tên miền ** thích ứng tên miền thực tế **:
- Các công ty có thư viện độc quyền
- Không thể tinh chỉnh tất cả chúng
- Cần bổ sung kiến ​​thức plug-and-play

**Kết quả**: Các mô hình nguồn mở nhỏ đạt được hiệu suất LLM gần như độc quyền đối với các tác vụ miền.

**Mẫu**:
1. Xây dựng miền KG (khái niệm, API)
2. Quản lý cơ sở trường hợp nhỏ (mẫu sử dụng)
3. Sử dụng KG để đảm bảo độ phủ sóng
4. Truy xuất cả hai, nạp vào LLM

Mẫu này khái quát hóa ngoài các lĩnh vực cho bất kỳ công việc kiến ​​thức chuyên ngành nào.

## 12. Công việc tương lai

Các tác giả đề cập:
- **Xây dựng KG tự động**: Giảm nỗ lực thủ công
- **Chuyển liên miền**: Kiến thức từ miền này sang miền khác
- **Học tập liên tục**: Cập nhật KG/trường hợp từ phản hồi
- **Đa phương thức**: Bao gồm sơ đồ, thông số kỹ thuật trong KG
- **Khả năng giải thích**: Hiển thị lý do truy xuất trường hợp

## 13. Những cân nhắc thực hiện

**Dành cho người hành nghề**:
1. **Đầu tư KG**: Chi phí một lần cho miền mô hình
2. **Bộ sưu tập tình huống**: Khai thác các kho lưu trữ hiện có hoặc viết ví dụ
3. **Tích hợp**: Bao bọc trong khung tác nhân hoặc sử dụng trình truy xuất độc lập
4. **Đánh giá**: Cần bộ thử nghiệm dành riêng cho từng miền

**Nỗ lực kỹ thuật**:
- Thiết lập ban đầu trung bình cao
- Ít phải bảo trì sau
- Có thể là dịch vụ/API cho nhiều dự án

## 14. Kết luận

DomAgent chứng minh rằng **kiến thức khái niệm có cấu trúc (KG) + mẫu trải nghiệm (CBR)** cho phép tạo mã theo miền cụ thể một cách hiệu quả mà không cần tinh chỉnh mô hình.

**Công thức chính**:
```
Kiến thức miền = Khái niệm (KG) + Mẫu (Trường hợp)
Tạo mã = ​​LLM + Truy xuất kép
```

**Kết quả**: 67% trên DS-1000 (cạnh tranh với các mẫu được tinh chỉnh), triển khai công nghiệp thành công.

**Bài học rút ra**: Đối với các miền chuyên biệt, hãy đầu tư vào **biểu diễn kiến ​​thức** (các trường hợp KG +) thay vì tùy chỉnh mô hình. Điều này dễ bảo trì hơn, di động hơn và tiết kiệm chi phí hơn.

**Ý nghĩa rộng hơn**: Cách tiếp cận "ưu tiên kiến ​​thức" này có thể mở rộng ra ngoài các lĩnh vực cho bất kỳ tình huống nào mà LLM thiếu kiến ​​thức cụ thể. Thay vì đào tạo lại, hãy **truyền đạt kiến ​​thức** dưới dạng có cấu trúc và truy xuất một cách thông minh.

DomAgent đại diện cho **con đường trung gian thực dụng** giữa RAG đơn giản và khả năng tinh chỉnh đắt tiền. Việc lựa chọn trường hợp theo hướng dẫn của KG đặc biệt thông minh - đảm bảo phạm vi phủ sóng có chất lượng mà không cần thư viện trường hợp lớn.
